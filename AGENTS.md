# AGENTS.md

Руководство для AI-агентов на уровне всего workspace.

## Структура репозитория

```text
apps/
  www/       # единое Astro-приложение kpshelkovo.online
  media/     # статическая 404 для media.kpshelkovo.online
packages/    # shared утилиты и UI
docs/        # документация, дизайн-гайды и ADR
ops/         # nginx и deploy-инфраструктура
scripts/     # build/deploy helper scripts
```

## Локальные инструкции

- Перед работой в `apps/www` обязательно читать `apps/www/AGENTS.md`.
- Для устойчивых названий, терминов, юрлиц и написаний сверяться с корневым `GLOSSARY.md`; новые подтвержденные термины добавлять туда.
- Данные новостей лежат в `apps/www/src/data/news`: авторы в `authors/*.yaml`, статьи в `articles/**/*.md`; при добавлении или редактировании новостей обязательно использовать skill `news-maker`.
- Данные статуса сервисов лежат в `apps/www/src/data/status/incidents/**/*.md`; перед добавлением или редактированием читать локальный `apps/www/src/data/status/incidents/AGENTS.md`.
- Перед загрузкой, заменой, удалением или проверкой публичных файлов в `kps-public` обязательно использовать skill `public-media-publisher`.
- Если меняются shared стили, примитивы ссылок или URL-утилиты, проверить все затронутые разделы в `apps/www`.
- Архитектурные решения искать и фиксировать в `docs/decisions`; список ADR лежит в `docs/decisions/README.md`.
- Публичные Markdown/llms-документы генерировать через `@shelkovo/markdown` AST API или app-helper поверх него; не собирать целый документ локальным `lines.join('\n')`.
- В публичных Markdown/llms-документах не использовать Markdown-таблицы; если нужна структурированная выдача, писать обычными списками.
- Не парься с форматированием кода - prettier все отформатирует автоматом при коммите
- В документации и ответах не использовать markdown-таблицы; писать списками.

## Роутинг news/status

- В `/status` добавляй доступность сервисов: электричество, вода, интернет, дамба, аварии, временные отключения, окна плановых или внеплановых работ, восстановление и фактическую длительность перебоя.
- В `/news` добавляй редакционные материалы и объявления с самостоятельной новостной ценностью: отчеты ОК Комфорт, тарифы, документы, собрания, голосования, строительные этапы, изменения правил, полезные инструкции и события для жителей.
- Если источник опубликован в Domyland или Telegram как `news`, маршрутизируй по смыслу, а не по названию канала: сообщение о временном отключении сервиса идет в `/status`.
- Если один источник содержит и сервисный перебой, и широкий контекст, добавляй отдельную status-запись для перебоя; новость делай только если остается самостоятельная польза вне таймлайна статуса.

## Видимые тексты

- Писать живым языком без канцелярита: активный залог вместо пассивов и отглагольных существительных (`залатали ямы`, а не `проведен ямочный ремонт`).
- Помогать читателю сканировать текст: выделять ключевые факты жирным, дробить длинный текст заголовками, ставить ссылки на связанные материалы.
- Эмоджи допустимы, когда уместны по тону и помогают восприятию.

## Упоминания людей

- Людей в материалах упоминать через `@person-id`, `@person-id:case` или `[видимый текст](@person-id)`.
- `@person-id` использовать, когда в тексте нужно показать каноническое имя человека в именительном падеже.
- `@person-id:case` использовать, когда нужно показать имя человека в другом падеже.
- `[видимый текст](@person-id)` использовать, когда видимым текстом должна остаться авторская фраза: роль, описание или часть предложения.
- Поддерживаемые падежи: `nom`, `gen`, `dat`, `acc`, `ins`, `prep`.
- Если падеж не указан, используется `nom`: `@kirill-shchemelinin` → именительный.
- Если в материале нужен другой падеж, сначала добавить соответствующую форму в данные человека, затем использовать ее в тексте: `@kirill-shchemelinin:gen`.
- Не использовать `[видимый текст](@person-id:case)`: в ссылочном формате грамматика живет в видимом тексте, а не в `name_cases`.
- Формы имени хранить в человеке как источник истины; не полагаться на автосклонение при рендере.
- Build должен проверять все `@person-id:case` в материалах: человек обязан существовать, а запрошенная форма склонения обязана быть заполнена.
- Нельзя молча fallback-ать на `nom`, если явно запрошен другой падеж и формы нет; build должен падать или выдавать явную ошибку.

## Правила Typescript

- по максимуму используй `readonly` для иммутабельности там, где она уместна
- для тривиальных helper-ов с единственным `return` предпочитай однострочные `const`-стрелки (`const fn = (): T => expr`) вместо `function fn() { return expr; }`
- не храни типы и интерфейсы вместе с кодом, лучше вынеси их в отдельный модуль
- не используй null вообще никогда, только undefined и optional свойства
- если параметр функции можно опустить, пиши `param?: T`, а не `param: T | undefined`; union с `undefined` оставляй для случаев, где аргумент обязателен семантически
- не писать `...(value ? { field: value } : {})` только ради optional-поля: если `undefined` допустим и отсутствие ключа не является контрактом, присваивай `field: value` напрямую

## Тесты

- Если ожидаемый результат не супермелкий и важна точная строка/структура, предпочитай `toMatchInlineSnapshot() / toMatchSnapshot()` (в зависимости от размера снапшота) вместо длинного `toBe(...)`/`toEqual(...)`
- Если в строке важны неразрывные пробелы, в snapshot показывай их видимым символом, например через test-helper, который заменяет `\u00A0` на `·`.
- Не генерируй тесты, которые проверяют случайную статику: точный текст заголовков/лейблов, полный generated markdown/llms output, большие snapshots или Tailwind-классы. Исключение — когда это явно публичный контракт, аудируемые данные или важная типографика/форматирование. Для generated discovery/llms проверяй наличие стабильных URL, схем, фидов и отсутствие внутренних путей; для UI — поведение, доступность и data/URL-контракты, а не классы.

## Правила workspace

- Общие стили и UI-примитивы выносить в `packages/ui`.
- Общие URL/build helper-утилиты выносить в `packages/url` или `scripts/`.
- При изменениях deploy-потока синхронно обновлять `.github/workflows/*` и `ops/nginx/*`.
- Если добавляется внешний ресурс или запрос вовне (`fetch`/`connect`, iframe, script, image, font, style, analytics, карты и т.п.), синхронно проверить и при необходимости обновить CSP в `ops/nginx/*`.
- Не запускать `pnpm dev` без явной просьбы: команда поднимает локальный dev-server на `:4321`.
