# Публичное хранилище файлов

Этот runbook описывает публикацию документов и скачиваемых вложений разделов сайта в S3. Архитектурные причины и границы решения зафиксированы в [ADR-021](../../docs/decisions/021-public-section-files-in-s3.md), а устройство публичного origin - в [ADR-019](../../docs/decisions/019-public-media-origin.md).

Правила для изображений пока не определены. Не применяй этот workflow к изображению без отдельного решения пользователя о пути, формате и способе публикации.

## Текущая конфигурация

- Провайдер: REG.RU Cloud Object Storage.
- Бакет: `kps-public`.
- Тип доступа: публичный.
- S3 API endpoint: `https://s3.regru.cloud`.
- Прямой virtual-hosted endpoint: `https://kps-public.s3.regru.cloud`.
- Канонический публичный origin: `https://media.kpshelkovo.online`.
- Nginx-конфиг origin: `ops/nginx/media-kpshelkovo-online.conf`.

Прямой endpoint провайдера нужен для диагностики, но не должен появляться в материалах сайта. Публичные ссылки имеют вид `https://media.kpshelkovo.online/<key>`.

Credentials для S3 уже настроены вне репозитория. Не записывай access key или secret key в документацию, команды, временные файлы, логи и ответы. `s5cmd` должен использовать существующую конфигурацию окружения.

Бакет публичный и доступен в обход nginx через endpoint провайдера. Не загружай секреты, персональные данные, внутренние материалы и любые файлы с ограниченным доступом.

## Структура ключей

S3 не хранит настоящие каталоги: пути ниже являются префиксами object key.

Первый сегмент обозначает раздел-владелец файла:

```text
<section>/<storage-specific path>
```

Сейчас определены префиксы `kb/` для файлов базы знаний и `815/` для промо-раздела `/815`. Внутренняя структура не обязана повторять вложенные URL раздела.

Текущие PDF промо-раздела `/815/regulation` используют недатированные ключи `815/regulation/<filename>.pdf`. Эти файлы не считаются историческим архивом и не получают дату снимка.

Исторические оферты используют схему:

```text
kb/ok/offer/<snapshot-date>/<slug>.pdf
```

Пример:

```text
kb/ok/offer/2026-07-07/shelkovo-park.pdf
```

Существующий `kb/ok/2026/plan.pdf` является опубликованным недатированным ключом. Не используй его как шаблон для документов с важной историей и не перезаписывай при появлении новой редакции.

Для новых ключей:

- используй строчные ASCII-буквы, цифры, дефисы и `/`;
- сохраняй корректное расширение файла;
- выбирай короткое устойчивое имя, которое не меняется вместе с редакционным заголовком;
- не создавай ключи `404.html` и `_media/*`: эти пути зарезервированы nginx;
- не рассчитывай на query string для версии или cache busting: media-origin его отбрасывает.

## Дата снимка

Для документов история редакций по умолчанию считается важной. Каждая редакция получает отдельный датированный ключ и после публикации не перезаписывается.

`<snapshot-date>` имеет формат `YYYY-MM-DD` и означает самую раннюю подтвержденную дату, когда проект видел конкретную редакцию:

1. Для нового файла используй дату его получения в `Europe/Moscow`.
2. Для переноса существующего материала можно использовать дату первого Git-коммита, который подтверждает эту редакцию.
3. Если историческую дату подтвердить нельзя, используй фактическую дату получения.

Не определяй дату снимка по filesystem creation date, S3 `Last-Modified` или автоматически по дате внутри документа. Если разные редакции претендуют на один key в течение одной даты, остановись и согласуй новый key вместо перезаписи.

Если материал уже ссылается на предыдущий снимок того же документа, сначала сравни переданный файл с опубликованным объектом. При одинаковом SHA-256 сохрани существующие дату, key и URL: одинаковые байты не образуют новую редакцию.

Недатированный изменяемый key допустим только по явному требованию, что история файла не нужна.

## Подготовка

Проверь наличие `s5cmd`:

```bash
s5cmd version
```

Убедись, что публикуешь именно переданный файл, и не изменяй его в рамках загрузки. Определи фактический MIME-тип и SHA-256:

```bash
file --brief --mime-type "<local-file>"
shasum -a 256 "<local-file>"
```

MIME-тип задавай явно при загрузке. Не определяй его только по расширению.

## Проверка существующего объекта

Перед каждой загрузкой проверь точный key:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" \
  head "s3://kps-public/<key>"
```

Если объект существует, скачай его во временный каталог и сравни SHA-256 с локальным файлом. Не полагайся на S3 ETag как на SHA-256: значение зависит от способа загрузки и multipart-разбиения.

- При одинаковом SHA-256 считай публикацию уже выполненной.
- При разном SHA-256 не перезаписывай объект и явно сообщи о конфликте.
- Ошибка `NotFound` означает, что можно переходить к загрузке.

## Загрузка

Для датированного неизменяемого key:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" cp \
  --no-clobber \
  --content-type "<mime-type>" \
  --cache-control "public, max-age=31536000, immutable" \
  "<local-file>" \
  "s3://kps-public/<key>"
```

Для текущих недатированных PDF промо-раздела `/815/regulation` используй тот же порядок с коротким кешем:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" cp \
  --no-clobber \
  --content-type "application/pdf" \
  --cache-control "public, max-age=300" \
  "<local-file>" \
  "s3://kps-public/815/regulation/<filename>.pdf"
```

`--no-clobber` является только дополнительной защитой от случайной перезаписи. В `s5cmd` проверка существования и загрузка не образуют атомарную операцию, поэтому не запускай конкурентные публикации одного key. Команда также может успешно пропустить существующий объект без заметного сообщения: после `cp` всегда повторно скачай объект и сравни SHA-256.

Nginx ограничивает клиентский `Cache-Control` на `media.kpshelkovo.online` пятью минутами независимо от долгого значения в метаданных S3. Долгий immutable cache относится к самому версионированному объекту и прямому endpoint провайдера.

## Проверка публикации

Публичный URL собирается без преобразования key:

```text
https://media.kpshelkovo.online/<key>
```

После загрузки проверь через этот URL:

- ответ `200`;
- ожидаемый `Content-Type`;
- размер ответа равен размеру локального файла;
- SHA-256 скачанного файла совпадает с локальным.

Не считай публикацию завершенной, пока не прошли все проверки. Добавляй URL в материалы сайта только после успешной проверки.

## Просмотр объектов

Показать все ключи раздела KB:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" \
  ls --show-fullpath "s3://kps-public/kb/*"
```

Показать метаданные одного объекта:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" \
  head "s3://kps-public/<key>"
```

## Замена и удаление

- Не заменяй датированный объект по тому же key.
- Не удаляй объект как побочный эффект уборки или переименования материала.
- Перед удалением найди известные ссылки в репозитории, покажи пользователю точный key и затронутые материалы, затем получи отдельное подтверждение.
- Учитывай, что внешние ссылки невозможно полностью найти по репозиторию; опубликованный URL по умолчанию считается долгоживущим.
- Если файл больше не должен быть публичным по юридической или security-причине, приоритет имеет ограничение доступа, но действие и затронутые ссылки нужно явно зафиксировать.

Удаляй только один буквальный key без wildcard:

```bash
s5cmd --endpoint-url "https://s3.regru.cloud" \
  rm --raw "s3://kps-public/<exact-key>"
```

Не используй для этой операции wildcard или `--all-versions`. После удаления проверь `s5cmd head` и публичный URL. Nginx может продолжать отдавать закешированный успешный ответ до 10 минут, поэтому немедленный `200` сам по себе не означает, что удаление не сработало.

## Итог операции

В финальном ответе укажи:

- S3 key;
- публичный URL на `media.kpshelkovo.online`;
- размер, MIME-тип и SHA-256;
- загружен новый объект или найден идентичный существующий;
- результат проверки публичного URL.

Не выводи credentials даже частично.
