# LLM-Friendly Estimate Details

## Problem Statement

Как дать LLM-агентам детальную смету 2026 так, чтобы они получали готовые досье по строкам сметы с составом, суммами, рисками и источниками, а не собирали ответ через сложные связи `work_items`, `resources`, `control_totals` и `source_refs`.

## Recommended Direction

Сделать новый публичный слой поверх текущего curated dataset: `estimate-details-agent-2026`. Текущие файлы `estimate-details-2026` остаются слоем извлечения и аудита, но перестают быть главным форматом для LLM-потребления.

Основная единица нового слоя - `row dossier` по `estimate_row_id`. Внутри: краткое описание строки, связанные услуги регламента, итоговая сумма, cost taxonomy, детальные `components`, проверки, спорные места и компактные источники. Агент должен уметь ответить на вопрос "что входит в эту строку и почему сумма такая" без открытия PDF.

`quote_items` нужно переосмыслить. Если это смысловая часть состава работ, это `components`. Если это табличная строка из PDF, это `evidence_items` или `source_rows`. Сейчас эти роли смешаны, поэтому данные выглядят как технический dump, хотя внутри уже есть полезная детализация.

## Key Assumptions to Validate

- [ ] Row-based dossier лучше для LLM, чем текущие тематические slices `materials.md`, `machines.md`, `labor.md`: проверить на 5 типовых вопросах и сравнить качество ответов.
- [ ] Из текущего dataset можно собрать хорошие `components` без повторного парсинга PDF: проверить на уборке, освещении и охране.
- [ ] Cost taxonomy стабильно ложится на все разделы: прогнать все `cost_bucket` через группы `direct_costs`, `indirect_costs`, `taxes`, `gross` и `other`.
- [ ] `source_refs` можно сократить в публичном слое без потери проверяемости: у каждого component/fact остается PDF, страница, fragment и исходная строка или цитата.

## MVP Scope

Это уже не MVP извлечения данных. Минимальная полезная версия нового слоя должна доказать, что агенту проще отвечать по строке сметы.

Входит:

- Новый публичный JSON contract с `rows[]` как главным интерфейсом.
- Per-row dossier для 3-5 representative строк из разных секций.
- `components[]` как смысловая детализация строки.
- `evidence[]` как компактный provenance без повторного dump-а всех источников.
- `cost_summary` с понятной таксономией затрат.
- `issues[]` для спорных мест, derived-расчетов и ручных проверок.
- Markdown-поверхность, которую агент может читать как справку, а не как внутренний dump.

## Not Doing (and Why)

- Не переписывать сразу весь extraction dataset: там уже много проверенной ручной работы.
- Не делать чистый facts graph главным API: агентам нужен ответный контекст, а не RDF-подобный конструктор.
- Не оставлять `quote_items` как публичное имя: оно маскирует полезную детализацию под provenance.
- Не оптимизировать первым номером под людей-аудиторов: главный потребитель сейчас LLM-агент.
- Не делать HTML UI: сначала нужен хороший JSON/markdown contract.

## Open Questions

- Финальное имя публичного слоя: `estimate-details-agent-2026`, `estimate-details-dossiers-2026` или `estimate-details-v2`?
- Должны ли `components` включать работы без суммы или только экономически оцененные элементы?
- Нужны ли per-row markdown URL вида `/815/regulation/details/rows/{estimate_row_id}.md`?
- Нужно ли после стабилизации нового слоя оставить старые `details/materials.md`, `details/machines.md`, `details/labor.md` как legacy или заменить их редиректами/индексами?
