# Идея 7: Каталог характеристик поселков

## Коротко

Характеристики поселков важны для Compare: поля инфраструктуры, общих пространств и модели обслуживания влияют на отображение, Markdown, рейтинг и обработку неизвестных значений. Сейчас метаданные полей копируются между схемой, рейтингом, таблицами и Markdown.

Идея: создать модуль каталога характеристик поселков, который владеет группами характеристик, порядком полей, подписями, отображением статусов, связями с рейтингом и правилом для неизвестных значений.

## Текущая картина

Факты о характеристиках распределены:

- схема задает допустимые поля и enum-значения;
- рейтинг задает веса и числовые соответствия;
- визуальные таблицы задают подписи, порядок и формулировки для показа;
- Markdown задает еще один набор подписей и формулировок статусов;
- схема обнаружения вручную перечисляет поля;
- документация объясняет обработку неизвестных значений и веса.

Ключевое продуктовое правило: неизвестное значение никогда не равно `no`. Это правило есть в математике рейтинга, визуальном отображении и пропусках в Markdown, но ни один модуль не владеет им целиком.

## Проблема

Добавление или изменение одного поля поселка требует ходить по многим файлам. Это слабая близость к одному месту изменения.

Интерфейс неявный: вызывающий код должен знать имя поля, группу, enum, подпись, порядок показа, вес в рейтинге, текст статуса, поведение неизвестного значения и поле публичной схемы. Реализация не намного сложнее этого разбросанного интерфейса, поэтому текущая форма поверхностная.

## Проверка удалением

Если удалить поле из схемы, ссылки в рейтинге, Markdown и визуальных таблицах не указывают обратно на одного владельца.

Если удалить массив подписей из таблицы, подписи Markdown и веса рейтинга все равно останутся отдельно.

Если меняется вес рейтинга, публичное объяснение и метаданные отображения не вынуждены измениться вместе с ним.

Сложность возвращается во многих местах вызывающего кода вместо того, чтобы концентрироваться в модуле каталога.

## Идея углубления

Создать модуль каталога характеристик поселков, который один раз описывает доменные характеристики:

- группа: инфраструктура, общие пространства, модель обслуживания;
- ключ поля;
- подпись;
- порядок показа;
- вид enum или соответствие статусов;
- правило для неизвестного значения;
- необязательный вес рейтинга и соответствие шкале;
- необязательные метаданные публичной видимости.

Схема, рейтинг, Markdown, визуальные таблицы и обнаружение могут потреблять факты каталога. Это не значит, что каждое поле должно участвовать в рейтинге; поля без рейтинга можно отметить явно.

## Что должно остаться за пределами

Каталог не должен сам считать итоговый рейтинг поселка. Модуль рейтинга может использовать метаданные каталога, но по-прежнему владеть составом формулы и логикой расстояния.

Каталог не должен рендерить целые таблицы или Markdown-страницы. Он дает метаданные характеристик; вызывающий код решает итоговую форму документа или интерфейса.

Каталог не должен трактовать неизвестное значение как `no`. Правило для неизвестного значения должно оставаться явным.

## Польза

Для вызывающего кода:

- Одно описание характеристики питает отображение, Markdown, рейтинг и проверки схемы.
- Добавление поля начинается с каталога, а тесты показывают пропущенных потребителей.
- Документация рейтинга может использовать структурированные метаданные вместо зеркального текста.

Для сопровождения:

- Расхождение подписей чинится в одном месте.
- Обработка неизвестных значений остается согласованной.
- Тесты покрытия полей становятся простыми и надежными.

## Тестовый эффект

Полезные тесты после углубления:

- у каждого поля схемы есть метаданные каталога;
- у каждого поля каталога есть группа и подпись;
- у каждого поля с рейтингом есть соответствие рейтинга;
- каждое соответствие рейтинга указывает на существующее поле каталога;
- визуальные таблицы и строки Markdown выводятся из порядка каталога;
- неизвестные поля пропущены или показаны как неизвестные, но никогда как `no`;
- `docs/rating` генерируется или проверяется по метаданным рейтинга.

## Риски и ограничения

- Каталог может стать слишком большим, если включить в него каждую фразу конкретного представления. Нужно отделить стабильные метаданные от текста конкретной поверхности.
- Визуальным Svelte-файлам может понадобиться более богатое форматирование, чем Markdown. Каталог должен давать факты, а не заставлять писать одинаковые формулировки везде.
- `docs/rating` сейчас внутренняя заметка. Если позже генерировать ее из каталога, нужно решить, остается ли точный текст ручным.

## Вопросы для обсуждения

- Какие подписи характеристик канонические: подписи таблиц, Markdown или `docs/rating`?
- Веса рейтинга должны жить в каталоге или в `RatingModel`, который ссылается на поля каталога?
- Каталог должен генерировать zod-схему или только проверять покрытие схемы?
- Как представить подписи для отдельных enum-значений, не делая интерфейс слишком широким?
