# Идея 6: Модуль данных Compare

## Коротко

Compare - центральная продуктовая поверхность тарифа 815. Сейчас `loadAllData()` возвращает сырые части, а вызывающий код заново выводит базовый поселок, места в рейтинге, оценки, сравнения, данные лент и факты для конкретных страниц.

Идея: углубить сборку данных Compare в один модуль, который владеет инвариантами сравнения поселков на этапе сборки.

## Текущая картина

`loadAllData()` возвращает:

- settlements;
- stats;
- comparisons;
- ratings.

Потом вызывающий код делает дополнительную работу:

- находит базовый поселок Шелково;
- расставляет поселки по местам;
- преобразует данные в строки для explorer;
- сортирует по рейтингу для fallback и Markdown;
- строит полную ленту;
- ищет данные страницы поселка;
- вычисляет или передает контекст расстояния;
- следит, чтобы карты сравнений совпадали со списком поселков.

Интерфейс отдает части, но настоящий инвариант - один согласованный набор данных Compare.

## Проблема

Модуль слишком поверхностный для своей роли. Вызывающий код должен знать порядок и связи между рейтингами, статистикой, сравнениями и списками поселков.

Это ухудшает близость к одному месту изменения. Ошибка в выборе базового поселка, покрытии рейтинговой карты или покрытии мест может по-разному проявиться в HTML, Markdown, JSON-лентах или состоянии Svelte.

## Проверка удалением

Если удалить `loadAllData()`, править придется много вызывающего кода, потому что он зависит от формы результата как от набора отдельных частей. Это показывает, что вызывающий код знает слишком много.

Если удалить `findBaseline()`, маршруты и Markdown должны заново реализовать поиск базового поселка.

Если удалить сборку данных страницы поселка на уровне маршрута, не остается единого владельца ответа "все факты, нужные странице поселка". Сложность возвращается в каждый маршрут.

## Идея углубления

Создать модуль данных Compare, который владеет полной сборкой на этапе build:

- сырые поселки из content collection;
- ровно один базовый поселок;
- покрытие рейтингами;
- покрытие сравнениями;
- покрытие местами;
- статистика;
- строки для explorer;
- строки полной ленты;
- факты для страницы каждого поселка;
- общие порядки сортировки для fallback и Markdown.

Точный интерфейс пока не проектируется. Важное направление: вызывающий код должен спрашивать у модуля данных нужную форму, а не заново собирать доменные связи.

## Что должно остаться за пределами

Svelte-explorer должен по-прежнему владеть состоянием браузерного взаимодействия. Модуль данных Compare не должен знать DOM или runtime карты.

Формула рейтинга остается отдельным кандидатом на модуль. Модуль данных должен связывать результаты рейтинга, но не обязательно владеть внутренностями формулы.

Сериализация лент может остаться в модулях лент, но они должны потреблять готовые факты набора данных, а не пересчитывать их.

## Польза

Для вызывающего кода:

- Маршруты, Markdown и JSON-endpoints используют одни и те же факты набора данных.
- Поиск базового поселка не повторяется.
- Новая поверхность Compare начинается с полного набора данных, а не с сырых частей.

Для сопровождения:

- Ошибки сборки данных концентрируются в одном модуле.
- Инварианты вроде "один базовый поселок" и "у каждого поселка есть рейтинг" могут падать раньше.
- Публичные поверхности реже расходятся, потому что используют одни и те же собранные факты.

## Тестовый эффект

Текущие тесты покрывают части: статистику, преобразование для explorer, поведение рейтинга. Более глубокий модуль данных позволит тестировать инварианты:

- ровно один базовый поселок;
- у каждого поселка есть рейтинг;
- у каждого поселка есть результат сравнения;
- у каждого поселка есть место;
- лента explorer и полная лента выводятся из одного набора данных;
- сортировка по умолчанию стабильна и общая;
- отсутствие данных страницы поселка имеет один режим ошибки.

Тесты маршрутов тогда сужаются до путей и поведения ответа.

## Риски и ограничения

- Этот модуль может стать слишком широким, если поглотит формулу рейтинга, метаданные характеристик, тарифную математику и схемы лент. Он должен связывать данные, а не проглатывать каждую реализацию Compare.
- Нужно сохранить `src/compare/lib/url.ts` как источник поведения URL/base для Compare по `apps/www/AGENTS.md`.
- Если реализовать идею слишком поспешно, может получиться гигантский объект с малой глубиной. Интерфейс должен отдавать полезные представления, а не просто еще более вложенные данные.

## Вопросы для обсуждения

- Какие представления набора данных действительно стабильны: главная Compare, страница поселка, лента explorer, полная лента, Markdown, `llms`?
- Построители лент должны жить за модулем данных Compare или оставаться отдельными адаптерами?
- Отсутствие базового поселка должно быть ошибкой сборки или явным отсутствующим состоянием?
- Какой порядок должен быть каноническим для публичных резервных списков?
