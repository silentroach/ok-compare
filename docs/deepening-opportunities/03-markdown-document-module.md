# Идея 3: Модуль Markdown-документов

## Коротко

ADR-008 закрепляет генерацию публичного Markdown через AST. Но код приложения все еще повторяет маленькие помощники: сериализация, строки `label: value`, разделы, фразы, ссылки на источники, пустые списки. В одном месте остался адаптер строкового документа: технически он парсит строки в AST, но сохраняет мышление "сначала строки".

Идея: добавить на уровне приложения модуль Markdown-документов с более выразительными примитивами для частых публичных документов. `mdast` остается деталью реализации для большей части кода приложения.

## Текущая картина

`@shelkovo/markdown` дает конструкторы `mdast` и сериализатор. Это хороший низкоуровневый модуль по ADR-008. Но модули приложения повторяют одни и те же приемы:

- сериализация документа из дочерних узлов;
- rows вида `label: value`;
- навигационные списки;
- пункт для пустого состояния;
- строки со ссылками;
- парсинг inline Markdown в элементы списка;
- заголовки разделов с содержимым.

`serializeMarkdownLineDocument` принимает `lines`, находит заголовки и разделы, потом парсит блоки. Формально публичный Markdown идет через AST-сериализатор, но исходная реализация остается массивом строк.

## Проблема

Текущий интерфейс Markdown на уровне приложения слишком близок к реализации `mdast`. Вызывающий код должен знать, когда использовать `md.heading`, когда `parseMarkdownFragment`, как не получить таблицу, как собрать строку `label: value`, как добавить пункт для пустого состояния.

Из-за этого модули остаются поверхностными: помощник `row` в Markdown Compare полезен только рядом с конкретным файлом, хотя такая же форма нужна в новостях, статусе, регламенте и `llms`. При изменении общего Markdown-стиля близость к единому месту изменения слабая.

## Проверка удалением

Если удалить локальные помощники `row/section/serialize` из одного Markdown-модуля, сложность не исчезает. Она переедет в соседние файлы или вернется прямо в код.

Если удалить `serializeMarkdownLineDocument`, у `reglament/detail-markdown.ts` нет полноценной замены на AST. Это показывает, что существующая граница недостаточно хорошо скрывает строковую реализацию.

Если убрать конструкторы `mdast` из вызывающего кода, большинство Markdown-модулей приложения не смогут выразить обычные документные шаблоны без копирования связующего кода.

## Идея углубления

Создать на уровне приложения модуль Markdown-документов, который поверх `@shelkovo/markdown` дает более выразительный интерфейс для публичных документов:

- заголовок документа и блоки метаданных;
- разделы с заголовками;
- строки с подписями;
- строки с URL;
- ссылки на источники;
- правило для пустых списков;
- навигационные блоки;
- элементы списка с inline Markdown, если они нужны.

Это не замена `@shelkovo/markdown`. Пакет остается универсальным AST-модулем. Модуль документов на уровне приложения знает только правила документов сайта, а не доменные данные.

## Что должно остаться за пределами

Доменный текст и выбор полей остаются в реализациях новостей, статуса, людей, регламента и Compare.

Модуль документов не должен превращаться в язык для любого возможного узла Markdown. Для редких случаев `mdast` код приложения может продолжать использовать `md` напрямую.

## Польза

Для вызывающего кода:

- Сгенерированные Markdown-файлы используют один документный словарь.
- Код приложения пишет доменные блоки, а не шаблонный код `mdast`.
- Новая публичная Markdown-поверхность начинается с общих документных примитивов.

Для сопровождения:

- Текст пустого состояния и формат строк меняются в одном месте.
- AST-безопасность и запрет таблиц остаются централизованными.
- Строковые документы можно постепенно мигрировать за одним адаптером.

## Тестовый эффект

Сейчас тесты рискуют проверять длинные snapshots, где основное место занимает раскладка строк. После углубления:

- тесты модуля документов один раз проверяют правила сериализации;
- тесты разделов проверяют стабильные URL и смысловые блоки;
- меньше тестов хранят полный сгенерированный Markdown в snapshot;
- регрессии вроде сломанной строки со ссылкой или поведения пустого списка падают на границе модуля документов.

## Риски и ограничения

- ADR-008 прямо говорит, что публичная поверхность пакета не должна становиться отдельным языком. Модуль приложения должен быть практичным и небольшим, а не универсальным DSL.
- Нельзя добавлять скрытую магию: если вызывающий код не понимает порядок разделов, интерфейс станет мутным.
- Нужно не нарушить правило `apps/www/AGENTS.md`: public Markdown генерируется через AST helpers или app-helper поверх них. Эта идея как раз про app-helper.
