# Миграция генерации Markdown на AST

## Статус

План работ. Это не ADR и не реализация.

## Контекст

ADR-004 принят: весь Markdown и Markdown-совместимый публичный текст сайта должен генерироваться через AST, а не ручной сборкой строк.

Основное решение:

- использовать mdast как представление документов;
- держать общий API генерации в `@shelkovo/markdown`;
- сериализовать Markdown через пакетный API, а не через локальные массивы строк;
- парсить уже написанные Markdown-фрагменты через пакетную вспомогательную функцию перед вставкой в итоговый документ;
- генерировать YAML frontmatter современным YAML-сериализатором и ставить кавычки только там, где они нужны;
- включить в миграцию `llms.txt` и `llms-full.txt`, потому что по структуре это Markdown-совместимые агентские документы;
- не сохранять текущий вид Markdown специально, если новая сериализация сохраняет смысл, ссылки и структуру;
- не добавлять ограничение импортов или архитектурный тест против ручной сборки строк на этом этапе.

## Общие правила для агентов

- Перед работой читать `AGENTS.md`, `apps/www/AGENTS.md` при изменениях в `apps/www`, ADR-004 и этот README.
- Выполнять только одну задачу за сессию, если пользователь явно не дал другое указание.
- Не запускать `pnpm dev` без явной просьбы.
- Не менять видимый текст намеренно. Если изменение текста стало необходимым, остановиться и согласовать отдельно.
- Не переписывать Markdown-to-HTML рендер из ADR-003: эта миграция только про генерацию Markdown-строк.
- Не добавлять защитное ограничение импортов, lint-правило или архитектурный тест, если задача явно этого не требует.
- После выполнения задачи поменять `Статус` в ее файле на `Выполнено` и добавить короткую заметку, если были отклонения от плана.

## Чекпоинты

После задачи 01:

- `@shelkovo/markdown` имеет протестированный API генерации Markdown;
- API умеет frontmatter, GFM, фрагменты Markdown и базовое экранирование;
- приложение еще не обязано быть мигрировано.

После задач 02-07:

- все сопровождающие `.md` страницы переведены на пакетный AST API;
- редакционные Markdown-фрагменты вставляются через парсинг, а не как обычный текст;
- старая вспомогательная функция frontmatter в приложении больше не нужна для сопровождающих `.md` страниц.

После задач 08-09:

- `llms.txt` и `llms-full.txt` тоже собираются через AST API;
- агентские тексты не имеют второго способа генерации Markdown-структуры.

После задачи 10:

- старая ручная вспомогательная функция frontmatter удалена или оставлена только при явно задокументированной причине;
- пакетная документация отражает API генерации;
- `pnpm test`, `pnpm typecheck` и `pnpm build` проходят или причина сбоя явно описана.

## Инварианты приемки

- Новая генерация использует `@shelkovo/markdown`, а не локальные сериализаторы в `apps/www`.
- Существующие публичные URL, заголовки документов, frontmatter-поля и JSON/discovery контракты сохраняют смысл.
- Вывод Markdown может отличаться по экранированию, кавычкам YAML, отступам списков и пустым строкам.
- Снимки тестов обновляются под новую нормальную сериализацию, а не под старую ручную раскладку строк.
- `llms.txt` остается `.txt`, но строится через тот же AST-подход.
