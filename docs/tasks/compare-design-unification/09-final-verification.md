# Task 09: Финальная проверка унификации

## Цель

Проверить, что compare действительно стал частью общей дизайн-системы сайта, а не просто потерял старые CSS overrides.

## Контекст

После Tasks 01-08 код должен быть технически чистым, но нужна финальная проверка маршрутов, тестов, сборки и архитектурных инвариантов.

## Что сделать

- Проверить отсутствие локальных `.ui-root-compare .ui-*` overrides по всему `apps/www/src`.
- Проверить отсутствие повторного `@import '@shelkovo/ui/styles.css'` в compare styles.
- Проверить compare pages на общий layout, breadcrumbs, footer, links, buttons и shell primitives.
- Проверить соседние разделы сайта, если менялся `packages/ui`.
- Собрать короткий итог: что изменено, что осталось локальным в compare, какие проверки выполнены.

## Чего не делать

- Не начинать новый refactor.
- Не менять визуальный слой в этой задаче, кроме мелких очевидных исправлений, найденных во время проверки.
- Не обновлять ADR.

## Критерии приемки

- Compare наследует shared primitives без section-level overrides.
- Локальные compare styles, если есть, относятся только к domain surfaces.
- Публичные data/API/LLM routes не изменены случайно.
- Тесты, typecheck и build проходят или каждый сбой явно объяснен.

## Проверка

- Запустить `pnpm --filter @shelkovo/www test`.
- Запустить `pnpm --filter @shelkovo/www typecheck`.
- Запустить `pnpm build`.
- При возможности вручную проверить `/815/compare/`, `/815/compare/rating/`, одну detail-страницу поселка, compare 404, `/news/`, `/status/`, `/815/regulation/`.

## Вероятно затронутые файлы

- Только если финальная проверка находит мелкие ошибки.
- Иначе task может не менять файлы.

## Зависимости

Зависит от Tasks 01-08.
