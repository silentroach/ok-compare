# Контекст для задач по полному регламенту

Этот файл нужно давать агенту вместе с одним конкретным task-файлом из `docs/reglament/full/tasks/`.

## Цель

Из `docs/reglament/original/full.pdf` нужно постепенно достать полезный слой фактов поверх уже существующего калькулятора `/reglament/`.

Главная цель не в том, чтобы заново разобрать всю смету, а в том, чтобы получить:

- машиночитаемые данные для LLM;
- проверяемые markdown-артефакты с ссылками на страницы PDF;
- основу для будущих отдельных страниц про имущество, услуги и характеристики поселков;
- критический слой: что в регламенте стоит проверять отдельно.

## Что уже есть

- `/reglament/` уже является калькулятором тарифа по смете 2026.
- Нормализованная смета лежит в `apps/www/src/data/reglament/estimate-2026.ts`.
- Машиночитаемый feed сметы уже есть по `/reglament/data/estimate-2026.json`.
- Исходные PDF детализации уже публикуются через `/reglament/original/{final,cleaning,landscaping,improvement,lighting,security,waste}.pdf`.
- Новый полный PDF лежит в `docs/reglament/original/full.pdf`.

## Принцип работы

Не пытаться за одну сессию разобрать весь PDF и сразу менять приложение.

Правильный порядок:

1. Сначала понять структуру PDF и страницы.
2. Потом извлекать отдельные наборы фактов в markdown.
3. Потом сопоставлять факты с уже существующей сметой.
4. Потом спроектировать JSON/LLM contract.
5. Только после этого менять код и страницы.

Каждая задача должна оставлять проверяемый документ в `docs/reglament/full/`, а не сразу код.

## Что брать из full.pdf

Брать в первую очередь:

- характеристики поселков: домовладения, площадь участков, разбивка по Village/Forest/Park/River;
- состав общего имущества: дороги, ливневки, зелень, лес, площадки, водоемы, ограждения, освещение, электрика, охрана;
- перечень услуг и периодичность работ;
- сопоставление услуг с текущими секциями сметы;
- хитрые или спорные расчетные допущения, если они влияют на понимание тарифа.

Не брать сейчас:

- методические основания и длинный список нормативной литературы;
- схемы/карты поселков;
- полное повторное извлечение сметных таблиц, которые уже легли в калькулятор;
- юридические выводы или обвинения.

## Тон

Для публичной части тон должен быть максимально простым.

Писать:

- `требует проверки`;
- `не найдено в явной строке сметы`;
- `частично сопоставлено`;
- `нужно сверить с первичными документами`.

Не писать:

- обвинения;
- сарказм;
- юридическую квалификацию;
- выводы, которых нет в PDF.

## Источник истины

Каждый факт должен иметь ссылку на источник:

- `pdf`: всегда `full` для фактов из полного регламента;
- `page`: страница PDF;
- `fragment`: раздел, приложение, строка или пункт;
- `quote`: короткая цитата, если она помогает проверить перенос;
- `note`: осторожное пояснение, если OCR или таблица неоднозначны.

Если значение взято из таблицы, но таблица плохо распознана, это нужно явно помечать как `требует визуальной сверки`.

## Предлагаемая будущая структура данных

Не реализовывать ее до task `090`, но держать в голове:

```ts
type FullReglamentFact = {
  id: string;
  category: 'village' | 'asset' | 'service' | 'assumption' | 'audit-note';
  title: string;
  summary?: string;
  source_refs: SourceRef[];
};
```

Отдельные сущности после извлечения могут стать:

- `villages[]`;
- `common_assets[]`;
- `services[]`;
- `service_to_estimate_map[]`;
- `audit_notes[]`.

## Возможные публичные страницы

Решение принимать только после задач извлечения.

Кандидаты:

- `/reglament/assets/` - общее имущество;
- `/reglament/services/` - услуги и периодичность;
- `/reglament/villages/` - характеристики поселков.

Главная `/reglament/` должна остаться калькулятором. На ней можно добавить только короткий блок ссылок на эти страницы.

## Проверка результата каждой задачи

Перед завершением задачи агент должен проверить:

- все факты имеют страницы PDF;
- нет данных из методических оснований и карт, если task явно не просил их брать;
- нет кода приложения, если task не `090`;
- итоговый markdown можно дать следующему агенту как источник.
