# Сверить данные регламента с PDF

Статус: done.
Дата: 2026-05-07.
Поверхность: `/reglament/`, `apps/www/src/data/reglament/estimate-2026.ts`, `docs/reglament/original/*.pdf`.

Результат: `docs/reglament/verification/estimate-2026-pdf-audit.md`, усилены тесты `estimate-2026.test.ts` и `calculate.test.ts`.

## Проблема

Данные сметы нормализованы вручную из PDF. Есть риск, что какая-то цифра, строка, источник или итог были распознаны или перенесены неверно.

Нужна отдельная сверка не UX-слоя, а фактической базы расчета.

## Что сверить

- Общий официальный итог из `final.pdf`: `221 264 198 ₽/год`.
- Официальный тариф: `902,07 ₽/сотка/мес`.
- Тарифицируемая площадь: `20 440,54 сотки`.
- Итоги всех 7 разделов из `final.pdf`.
- Все 19 строк в `estimate-2026.ts`: название, годовая сумма, вклад в тариф, источник PDF/страница/фрагмент.
- Все исходные параметры строк: объем, кратность, цена/ставка, если они есть.
- Все статьи `CostBreakdown`: зарплата, зарплата машинистов, ФОТ, машины, материалы, подрядчики, страховые, общеэксплуатационные, прибыль, УСН, доходы, НДС, итог с НДС.
- Сходимость сумм строк с итогами разделов.
- Сходимость итогов разделов с общим итогом.
- Сходимость расчетного движка с baseline feed.
- Все caveats и `требует проверки`, особенно строку ремонта дорог/площадок против детализации благоустройства.

## Методика

1. Для каждого PDF выписать контрольную таблицу: страница, строка/раздел, PDF-значение, значение в `estimate-2026.ts`, статус сверки.
2. Для каждой строки проверить не только итог, но и состав затрат, если он есть в детализации.
3. Отдельно отметить места, где PDF не дает прямой детализации или название строки не совпадает.
4. Проверить формулы: `ФОТ`, страховые, общеэксплуатационные, прибыль, УСН, доходы, НДС, `gross`, `tariff_per_sotka_month`.
5. Если найдена ошибка, исправить данные и добавить/обновить тест, который бы ловил регрессию.

## Результат task-а

Должен появиться один из вариантов:

- файл сверки, например `docs/reglament/verification/estimate-2026-pdf-audit.md`;

- либо оба результата.

## Критерии приемки

- Для каждой секции и строки явно указано: сверено, не сверено или требует внешнего подтверждения.
- Все суммы строк сходятся с секционными итогами или имеют объясненный caveat.
- Все секционные итоги сходятся с общим итогом.
- Спорная строка `improvement-road-surface-repair` отдельно разобрана и не маскируется как подтвержденная.
- После правок проходят `pnpm --dir apps/www test` и `pnpm --dir apps/www typecheck`.
