# Калькулятор тарифа по смете 2026

Статус: продуктовая спецификация для реализации.
Дата: 2026-05-07.
Зона реализации: `apps/www`, новый публичный раздел по смете регламента.

## Проблема

Смета регламента ОК читается как набор PDF-таблиц, где важные величины смешаны с нормативными основаниями, ресурсными ведомостями и техническими строками. Собственнику трудно понять, какие параметры формируют тариф `902,07 ₽/сотка/мес`, и что изменится, если оспорить частоту работ, объем, ставку или подрядную цену.

Нужно сделать нейтральный интерактивный калькулятор: вся смета видна на странице, пользователь может менять значимые поля и сразу видеть новый тариф за сотку в месяц.

## Рекомендованное направление

Делать не PDF-таблицу и не разоблачительный материал, а аккуратный финансовый инструмент. Главная единица интерфейса - `₽/сотка/мес`, а не годовая сумма и не цена участка пользователя. Годовые суммы нужны в данных, деталях и проверках, но не должны быть главным визуальным числом.

Основной экран строится из последовательных секционных таблиц: `Перемещение мусора`, `Уборка территории`, `Озеленение`, `Благоустройство`, `Охрана`, `Организация работы с РО по вывозу мусора`, `Освещение и электроснабжение`. Дерево секций не нужно: оно добавит когнитивной вложенности. Каждая секция сама является плоским блоком с таблицей и компактным итогом секции.

Тон страницы максимально нейтральный. Допустимы только фактические бейджи вроде `крупная статья`, `подрядная услуга`, `ручной труд`, `машино-часы`, `требует проверки`. Саркастические подписи и оценочные формулировки не использовать.

## Источники

- `docs/reglament/original/final.pdf` - итоговая публичная смета, источник официального общего тарифа и секционных итогов.
- `docs/reglament/original/cleaning.pdf` - детализация уборки территории.
- `docs/reglament/original/landscaping.pdf` - детализация озеленения.
- `docs/reglament/original/improvement.pdf` - детализация благоустройства.
- `docs/reglament/original/lighting.pdf` - детализация освещения и электроснабжения.
- `docs/reglament/original/security.pdf` - детализация охраны.
- `docs/reglament/original/waste.pdf` - детализация организации вывоза и перемещения мусора.

## Базовые числа

- Тарифицируемая площадь: `20 440,54 сотки`.
- Официальный итог: `221 264 198 ₽/год`.
- Официальный тариф: `902,07 ₽/сотка/мес`.
- Формула тарифа: `годовой итог / 20 440,54 / 12`.

Важная деталь: `final.pdf` сходится не с локальной строкой `Сметная стоимость`, а с полной строкой `Доходов всего` из калькуляции, умноженной на НДС 5%. Например, для уборки территории: `151 439 865 × 1,05 = 159 011 858`. Поэтому расчетная модель должна учитывать не только прямые затраты, страховые, общеэксплуатационные и прибыль, но и налог по УСН внутри расчетной прибыли, а затем НДС 5%.

## Расчетная модель

Глобальные параметры по умолчанию:

| Параметр                     |          Значение | Комментарий                           |
| ---------------------------- | ----------------: | ------------------------------------- |
| Площадь тарификации          | `20 440,54 сотки` | из `final.pdf`                        |
| Страховые взносы             |           `30,2%` | от ФОТ                                |
| Общеэксплуатационные расходы |             `70%` | от ФОТ                                |
| Прибыль                      |             `40%` | от ФОТ                                |
| УСН                          |             `15%` | от прибыли, остающейся в распоряжении |
| НДС                          |              `5%` | применяется к доходам всего           |

Базовая формула для позиции:

```text
fot = основная зарплата + зарплата машинистов
direct = fot + машины + материалы + подрядчики
insurance = fot * 0.302
overhead = fot * 0.70
profit = fot * 0.40
usn = profit * 0.15
income = direct + insurance + overhead + profit + usn
gross = income * 1.05
tariff = gross / 20 440,54 / 12
```

Для подрядных или материальных позиций без ФОТ страховые, общеэксплуатационные, прибыль и УСН обычно равны нулю. Это поведение должно быть задано явно в данных позиции, а не угадываться только по нулевым значениям.

## Модель данных

Данные не парсить из PDF на клиенте. PDF-парсинг слишком нестабилен для таблиц такого качества. Нужен нормализованный статический слой в `apps/www/src/data/reglament/estimate-2026.ts` или аналогичном файле.

Минимальная структура сущностей:

| Сущность            | Смысл                                                                       |
| ------------------- | --------------------------------------------------------------------------- |
| `Estimate`          | год, площадь, глобальные коэффициенты, секции, официальный итог             |
| `EstimateSection`   | название секции, итог из `final.pdf`, список строк                          |
| `EstimateRow`       | пользовательская строка работы или сгруппированная подстрока                |
| `EstimateSourceRef` | PDF, страница, фрагмент/закладка, опциональная заметка                      |
| `EditableField`     | поле, которое можно менять в UI                                             |
| `CostBreakdown`     | ФОТ, машины, материалы, подрядчики, страховые, накладные, прибыль, УСН, НДС |

Каждая строка должна иметь стабильный `id`, человекочитаемое название, исходные значения, формулу пересчета, ссылки на источник и набор редактируемых полей. Служебные ресурсные строки вроде спецодежды, мыла и инвентаря можно группировать под родительской строкой, если отдельное редактирование не дает пользователю понятной ценности.

## UI

### Верх страницы

- Заголовок: `Калькулятор тарифа по смете 2026`.
- Короткое объяснение: это интерактивная версия сметы регламента, итог считается в `₽/сотка/мес`.
- Компактная итоговая панель: официальный тариф, текущий тариф, разница.
- Поле `мой участок` не делать в первом релизе. Можно оставить как открытый вопрос на будущее.

### Sticky-панель

Sticky должна быть супер-компактной.

Десктоп:

```text
902,07 ₽/сотка/мес -> 861,20 ₽    -40,87 ₽    Сбросить
```

Мобайл:

```text
861,20 ₽/сотка/мес   -40,87 ₽   Сброс
```

Панель не должна превращаться в карточку-дашборд. Это строка текущего состояния, а не отдельный экран.

### Секционные таблицы

Каждая секция рендерится отдельной таблицей с компактным итогом секции над таблицей или в первой строке секции.

Основные колонки:

| Колонка       | Смысл                                                                         |
| ------------- | ----------------------------------------------------------------------------- |
| Работа        | короткое название позиции                                                     |
| База          | объем или расчетная база, например `81 778 м²`, `506 светильников`, `4 поста` |
| Кратность     | количество раз/дней/месяцев в год                                             |
| Цена/ставка   | ставка, цена услуги или ставка ресурса                                        |
| Вклад в тариф | `₽/сотка/мес` по позиции                                                      |
| Изменение     | дельта к официальной строке после правок                                      |
| Детали        | раскрытие расчетного состава и источников                                     |

Годовую сумму не показывать как основную колонку. Ее можно показывать в раскрытии строки и отдавать в machine-readable data.

### Редактирование

В обычном режиме пользователь меняет только понятные поля: объем, кратность, цену/ставку, включение позиции. В деталях можно дать экспертные поля: трудозатраты, машино-часы, ФОТ, материалы, подрядчики и коэффициенты строки.

Все изменения должны пересчитываться мгновенно без запроса на сервер. Сброс должен возвращать официальную смету.

### Визуальный стиль

- Следовать `docs/design/design-code-shelkovo.md`.
- Почти белый фон, чистые белые поверхности, глубокий зеленый текст и действия.
- Не делать многоуровневые карточки, стеклянные панели, тяжелые тени, желтые/бумажные фоны.
- Таблицы должны быть опрятными и плотными, с четкими разделителями.
- Допустим один плоский контейнер на секцию, без вложенных островов и декоративной иерархии.

## Техническая архитектура

Предпочтительный вариант: Astro-страница с одним плоским клиентским контроллером на vanilla TypeScript. Не добавлять Svelte/React только ради калькулятора, если обычный DOM-контроллер справится.

Примерная структура:

| Файл                                                                 | Роль                               |
| -------------------------------------------------------------------- | ---------------------------------- |
| `apps/www/src/data/reglament/estimate-2026.ts`                       | нормализованные данные сметы       |
| `apps/www/src/lib/reglament/schema.ts`                               | типы и JSON Schema helpers         |
| `apps/www/src/lib/reglament/calculate.ts`                            | чистый расчет без DOM              |
| `apps/www/src/lib/reglament/discovery.ts`                            | JSON payload, API catalog, OpenAPI |
| `apps/www/src/lib/reglament/llms.ts`                                 | короткий и полный агентный обзор   |
| `apps/www/src/lib/reglament/routes.ts`                               | маршруты раздела                   |
| `apps/www/src/pages/reglament/index.astro`                           | HTML-страница калькулятора         |
| `apps/www/src/pages/reglament/index.md.ts`                           | markdown companion                 |
| `apps/www/src/pages/reglament/data/estimate-2026.json.ts`            | machine-readable data              |
| `apps/www/src/pages/reglament/llms.txt.ts`                           | короткий agent overview            |

Если интерактивность станет слишком сложной для DOM-контроллера, допустим один Svelte island верхнего уровня. Вложенные острова не использовать.

## Agent Discovery

AI-поверхности являются обязательной частью фичи, не постфактум.

Нужно отдать агентам:

- полный нормализованный список секций и строк;
- официальные значения и текущие формулы;
- все исходные поля, даже если UI показывает не все;
- источники по каждой строке: PDF, страница, закладка/секция;
- секционные итоги из `final.pdf`;
- вычисленные вклады в `₽/сотка/мес`;
- caveats о качестве PDF и необходимости проверки специалистом;
- JSON Schema и OpenAPI-like описание статического JSON;
- markdown companion для людей и LLM;
- обновление корневого `/.well-known/api-catalog`.

Публичные discovery-маршруты:

| Route                                           | Назначение                     |
| ----------------------------------------------- | ------------------------------ |
| `/reglament/`                                   | HTML-калькулятор               |
| `/reglament/index.md`                           | markdown companion             |
| `/reglament/data/estimate-2026.json`            | основной JSON feed             |
| `/reglament/llms.txt`                           | краткий обзор для агентов      |
| `/reglament/llms-full.txt`                      | полный обзор для агентов       |
| `/reglament/schemas/estimate-2026.schema.json`  | схема JSON feed                |
| `/reglament/openapi/estimate-2026.openapi.json` | описание статических endpoints |
| `/reglament/.well-known/api-catalog`            | section-level API catalog      |

## Не делаем сейчас

- Расчет платежа для конкретной площади участка.
- Сценарии с share-link.
- Авторизационные или пользовательские сохранения.
- Автоматический runtime-парсинг PDF.
- Саркастические комментарии внутри калькулятора.
- Полную копию исходных PDF-таблиц в UI.

## Открытые вопросы

- Финальный публичный route: `/reglament/` или `/tariff/`. Текущая рекомендация: `/reglament/`, потому что источник и раздел называются регламентом.
- Публиковать ли исходные PDF как статические файлы сайта. Если да, их нужно положить в `apps/www/public/reglament/original/` или добавить build-copy.
- Нужно ли в экспертном режиме редактировать глобальные коэффициенты, или достаточно редактировать строки.
- Насколько мелко дробить ресурсные строки спецодежды и инвентаря: отдельными строками в данных или только в деталях родительских работ.
