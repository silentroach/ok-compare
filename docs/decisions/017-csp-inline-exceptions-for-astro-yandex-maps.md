# ADR-017: CSP с inline-исключениями для Astro и Яндекс Карт

## Статус

Принят, заменяет [ADR-005](005-csp-inline-script-policy.md)

## Дата

2026-07-01

## Контекст

ADR-005 зафиксировал цель убрать `script-src 'unsafe-inline'` для исполняемых скриптов: проектный клиентский код был вынесен в `apps/www/src/scripts/site-runtime.ts`, а `apps/www/astro.config.ts` получил `vite.build.assetsInlineLimit: 0`.

После развития сайта текущим источником истины для CSP стали `ops/nginx/kpshelkovo-online.conf` и `ops/nginx/CSP.md`. В них боевая политика снова содержит `script-src 'unsafe-inline' 'unsafe-eval'`:

- `unsafe-inline` нужен, потому что Astro генерирует inline bootstrap-скрипты и загрузчики islands прямо в HTML;
- `unsafe-eval` нужен JS API Яндекс Карт v3 для работы векторного движка при парсинге тайлов;
- `style-src 'unsafe-inline'` остается нужен для сгенерированных inline-стилей и стилей сторонних виджетов.

При этом исходная граница безопасности остается важной: проектный исполняемый код не должен расползаться по raw inline-скриптам и inline-обработчикам событий.

## Решение

Текущая боевая CSP может содержать `script-src 'unsafe-inline'` и `script-src 'unsafe-eval'` только как документированные исключения для текущих генераторов и интеграций.

`script-src-attr 'none'` остается обязательным: inline-обработчики событий вроде `onclick="..."` не разрешаются.

Исполняемый код, который пишет проект, по-прежнему нужно держать во внешних обработанных модулях: `src/scripts/*`, компонентных обработанных `<script>` без `is:inline` или других собранных ассетах. Новый raw inline-скрипт требует отдельной причины и синхронного обновления CSP-документации.

`ops/nginx/CSP.md` остается обязательным пояснением для каждого широкого или стороннего источника CSP. При изменении `Content-Security-Policy` нужно синхронно менять `ops/nginx/kpshelkovo-online.conf` и `ops/nginx/CSP.md`.

Практическая цель политики меняется с «в `script-src` нет `unsafe-inline`» на более точную для текущего сайта:

- нет неучтенных raw inline-скриптов проекта;
- нет inline-обработчиков событий;
- сторонние источники перечислены узко и по директивам;
- `unsafe-inline` и `unsafe-eval` остаются явно принятыми временными исключениями, а не нормой для нового кода.

## Рассмотренные альтернативы

### Вернуть hash-based `script-src` для всех inline-фрагментов

Плюсы:

- можно снова убрать `unsafe-inline` из `script-src`;
- Lighthouse получает более строгую CSP.

Минусы:

- Astro и islands генерируют inline-фрагменты, чувствительные к версии фреймворка, директивам и итоговому HTML;
- каждое изменение bootstrap-фрагментов требует синхронного обновления nginx;
- ошибка хэша ломает клиентскую навигацию или islands на боевом сайте.

Отклонено сейчас: хэши остаются хорошим направлением усиления, но текущая стоимость поддержки выше практической пользы.

### Ввести nonce на уровне nginx

Плюсы:

- можно точечно разрешать inline-скрипты на каждом ответе;
- модель лучше подходит динамическому HTML.

Минусы:

- сайт собирается статически, а nginx должен был бы менять HTML и CSP на каждый ответ;
- это усложняет деплой и кеширование ради статического сайта;
- nonce не решает `unsafe-eval`, нужный Яндекс Картам.

Отклонено: для текущей статической архитектуры nonce непропорционально усложняет обслуживающий слой.

### Убрать Яндекс Карты или перейти на iframe-only карты

Плюсы:

- можно убрать `unsafe-eval`, если JS API v3 больше не используется;
- CSP стала бы заметно уже.

Минусы:

- интерактивные сценарии Compare завязаны на JS API Яндекс Карт;
- iframe-only вариант меняет продуктовый UX и техническую модель карт;
- это продуктовое решение, а не безопасная правка документации.

Отклонено в рамках этого ADR: карту можно менять отдельным продуктово-техническим решением.

## Последствия

- ADR-005 больше не описывает текущую боевую CSP и остается историческим контекстом.
- Lighthouse `csp-xss` может продолжать ругаться на `unsafe-inline`, `unsafe-eval` или сторонние источники; это принятое ограничение текущих интеграций, а не скрытая регрессия.
- Перед удалением `script-src 'unsafe-inline'` нужно доказать, что inline-фрагменты Astro можно надежно хэшировать, вынести или иначе разрешить без хрупкого ручного сопровождения.
- Перед удалением `script-src 'unsafe-eval'` нужно убрать JS API Яндекс Карт v3, изменить режим карт или подтвердить, что текущая версия API больше не требует eval.
- Усиление `style-src` без `unsafe-inline` остается отдельным будущим направлением и требует проверки Astro, Tailwind output и сторонних виджетов.
