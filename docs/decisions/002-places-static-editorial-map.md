# ADR-002: Предсобранная редакционная карта мест

## Статус

Принят

## Дата

2026-05-13

## Контекст

Жителям КП Шелково и их гостям нужен быстрый способ выбрать полезное место рядом с поселком: где поесть, куда съездить с детьми, где купить нужное, куда обратиться по здоровью или что показать гостям.

Обычные карты уже решают задачу поиска по названию и построения маршрута, но плохо отвечают на локальный вопрос: что из ближайшего действительно полезно жителю Шелково и почему это место стоит открыть. Сайту нужен не дубль Яндекс Карт, а короткий редакционный слой поверх карты: отобранные места, понятные сценарии, ссылки на маршрут и внешние источники.

`Шелково Онлайн` остается сайтом со статической сборкой. Публичные разделы должны быть полезны без JavaScript, индексироваться поисковиками и оставаться удобными для агентов. В `/815/compare` уже используется похожая схема: данные собираются во время сборки, страница получает заранее собранный HTML-блок, а интерактивный Svelte-слой работает поверх статического JSON.

## Решение

Сделать раздел `/places/` как предсобранную редакционную карту мест вокруг КП Шелково.

Первый продуктовый фокус - редакционный топ-20 и сценарный выбор, а не полный каталог всего рядом. Раздел должен помочь человеку быстро выбрать место под ситуацию, а не заменить внешние карты.

Главная страница `/places/` должна совмещать:

- карту с маркерами для пространственного понимания;
- список карточек для быстрого выбора, особенно на мобильном;
- фильтры по типам мест: `cafe`, `restaurant`, `shop`, `clinic`, `doctor`, `pharmacy`, `kids`, `sport`, `poi`, `service`;
- фильтры по сценариям: `with-guests`, `with-kids`, `quick-food`, `on-the-way-home`, `health`, `walk`, `useful-nearby`;
- быстрые ссылки на внешнюю карту, сайт, телефон или другой проверочный источник, если они есть.

Тип отвечает на вопрос "что это". Сценарий отвечает на вопрос "зачем мне это сейчас". Сценарии являются обязательной частью продукта, потому что именно они отличают раздел от обычного справочника.

Для каждого места нужен стабильный адрес страницы. Общая страница может показывать краткую карточку, но подробное описание, фото, источники и практические заметки должны иметь отдельную страницу места. Это делает места удобными для отправки ссылкой, поиска и машиночитаемых поверхностей.

## Требования к статической сборке

Раздел должен быть устроен так:

- данные мест загружаются во время сборки;
- общая страница содержит заранее собранный HTML со списком карточек, минимум для первого контентного среза;
- интерактивная карта и фильтры улучшают опыт, но не заменяют HTML-контент;
- интерактивный слой получает данные из статического JSON-адреса;
- HTML, JSON и будущие машиночитаемые поверхности должны описывать одну и ту же версию данных;
- если раздел становится публичным контрактом, нужно добавить сопровождающую Markdown-страницу, JSON-ленту, схему данных или включение в `llms.txt` по аналогии с другими разделами.

JavaScript может скрывать или улучшать запасной HTML-блок после загрузки интерактивного слоя, но страница без JavaScript должна оставаться полезной: человек должен видеть названия мест, краткие описания, типы, сценарии и ссылки на маршруты или источники.

## Не решаем этим ADR

Этот ADR не выбирает конкретную карту, поставщика картографических тайлов или JavaScript-библиотеку. При реализации любой внешний `script`, `style`, `connect`, `image`, `font`, iframe или адрес тайлов должен пройти отдельную проверку CSP в `ops/nginx/*`.

Этот ADR также не вводит маршруты, готовые прогулки, рейтинги мест или автоматический импорт из внешних карт. Эти функции можно рассматривать позже, если первый редакционный слой окажется полезным.

## Рассмотренные альтернативы

### Только ссылки на внешние карты

Плюсы:

- минимальная реализация;
- нет интерактивной карты на сайте;
- меньше ответственности за интерфейс.

Минусы:

- сайт не отвечает на вопрос, почему место полезно жителю Шелково;
- сценарии и редакционный отбор остаются слабо выраженными;
- данные хуже индексируются и хуже переиспользуются агентами.

Отклонено: внешние карты нужны как источник и маршрут, но не как единственный интерфейс раздела.

### Карта только на JavaScript

Плюсы:

- проще сделать визуально интерактивный интерфейс;
- меньше дублирования в HTML;
- можно быстрее прототипировать фильтры и маркеры.

Минусы:

- страница без JavaScript становится пустой или почти пустой;
- поисковая индексация и поверхности для агентов деградируют;
- нарушается статически собранный характер сайта.

Отклонено: интерактивность должна улучшать заранее собранный контент, а не быть единственным способом получить данные.

### Полный каталог всех мест рядом

Плюсы:

- выглядит масштабнее;
- потенциально закрывает больше поисковых запросов;
- может стать справочником района.

Минусы:

- быстро превращается в слабую копию Яндекс Карт;
- требует постоянной модерации и обновлений;
- усложняет старт и размывает редакционную ценность.

Отклонено для первой версии: первый срез должен доказать пользу отбора, а не объем базы.

### Общая страница без отдельных страниц мест

Плюсы:

- меньше маршрутов и шаблонов;
- все видно на одной странице;
- проще первая сборка.

Минусы:

- сложнее делиться конкретным местом;
- фото, источники и практические заметки перегружают общий список;
- хуже формируются поисковые поверхности и материалы для агентов по отдельному месту.

Отклонено: стабильные страницы мест лучше подходят для редакционной базы и будущих упоминаний.

## Последствия

- `/places/` становится отдельным публичным разделом сайта.
- Первую версию нужно начинать с ограниченного, отобранного набора мест, а не с попытки покрыть все вокруг.
- Интерфейс должен проектироваться как карта плюс список, а не как карта вместо списка.
- Мобильный сценарий должен позволять выбрать место через список без обязательного взаимодействия с картой.
- Данные для HTML, JSON и поверхностей для агентов должны собираться из одного источника.
- При добавлении внешних ресурсов для карты нужно синхронно проверять CSP и nginx-конфиги.
- Модель данных мест фиксируется отдельно в [ADR-003](003-places-markdown-content-model.md).
