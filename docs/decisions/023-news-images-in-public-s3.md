# ADR-023: Фотографии новостей в публичном S3

## Статус

Принят

## Дата

2026-07-23

## Контекст

Новостям нужны фоторепортажи с несколькими крупными снимками. Хранить такие JPEG в Git неудобно: бинарные файлы раздувают репозиторий и его историю. При этом публикация только внешней ссылки ненадежна, а исходники современных телефонов на 12-48 Мп избыточны для сайта и могут содержать непредназначенные для публикации EXIF, IPTC или XMP-метаданные.

ADR-021 определил общее публичное S3-хранилище для документов, но оставил правила изображений открытыми. Для первого конкретного сценария нужны устойчивые ключи, достаточно качественная публикационная копия, размеры для предотвращения CLS и адаптивные производные для веб-страницы.

## Решение

- Хранить подготовленные JPEG-копии новостных фотографий в бакете `kps-public` под ключами `news/YYYY/MM/<article-slug>/<semantic-name>.jpeg`.
- Перед загрузкой применять ориентацию снимка, не увеличивать изображение и ограничивать длинную сторону 2560 пикселями. Сохранять progressive JPEG в sRGB с quality 90 и цветовой субдискретизацией 4:2:0.
- Лимит 2560 пикселей рассчитан на текущую максимальную ширину галереи 1132 CSS-пикселя и дает запас примерно для 2x DPR без хранения полного телефонного исходника.
- По умолчанию удалять все EXIF, IPTC, XMP, комментарии, thumbnails и другие вложенные метаданные.
- Сохранять только `GPSLatitude`, `GPSLatitudeRef`, `GPSLongitude` и `GPSLongitudeRef`, если место съемки само является публичным предметом материала и координаты решено опубликовать осознанно. Данные устройства, автора, времени съемки и другие GPS-поля не переносить.
- Считать каждый такой ключ неизменяемым. Не перезаписывать опубликованный объект; при конфликте выбрать новый согласованный ключ.
- Загружать подготовленный JPEG с `Content-Type: image/jpeg` и `Cache-Control: public, max-age=31536000, immutable`.
- Публиковать полноразмерную подготовленную копию только через `https://media.kpshelkovo.online/<key>`.
- Хранить во frontmatter новости канонический URL подготовленной копии, ее ширину, высоту, `alt` и опциональную подпись.
- Разрешать для `photos[].url` только origin `media.kpshelkovo.online`, префикс `/news/` и URL без query string или fragment.
- Передавать обложку и все фотографии новости в `NewsArticle.image` JSON-LD; обложку ставить первой, одинаковые URL не дублировать.
- Показывать фото через Astro Image Service. Статическая сборка получает подготовленный JPEG из S3 и генерирует адаптивные WebP-варианты; ссылка с изображения ведет на полноразмерную подготовленную копию.
- Сохранять `cover` отдельным локальным build asset до отдельного решения о внешних обложках и social preview.
- Не распространять этот workflow автоматически на изображения базы знаний, регламента и других разделов.

## Рассмотренные альтернативы

### Хранить фотографии рядом с Markdown в Git

Astro оптимизирует локальные изображения без дополнительной конфигурации, но полноразмерные фоторепортажи заметно увеличивают Git-репозиторий и остаются в его истории после удаления из актуального материала.

Отклонено для новостных фоторепортажей; локальный `cover` пока сохраняется как отдельный существующий сценарий.

### Хранить исходники камеры без изменений

Это сохраняет максимум исходных данных, но расходует лишнее место, замедляет чистую сборку и может случайно опубликовать модель устройства, время съемки, координаты и другие скрытые метаданные.

Отклонено: публичное S3 хранит подготовленную для сайта копию, а архивирование исходников при необходимости должно быть отдельным непубличным процессом.

### Ограничить длинную сторону 1600 пикселями

Такой файл был бы меньше, но не давал бы 2x-запаса для снимка во всю ширину текущей галереи и быстрее потерял бы актуальность на экранах с высокой плотностью пикселей.

Отклонено в пользу 2560 пикселей по длинной стороне; более мелкие адаптивные варианты по-прежнему создает Astro.

## Последствия

- Новостные фотографии не попадают в Git, но остаются доступны по долгоживущим URL.
- S3-копия является качественным web-master, а не архивным оригиналом камеры.
- Сборка новостей зависит от доступности разрешенных S3-фотографий и тратит дополнительное время на генерацию производных.
- `photos` в публичном news API содержит `width` и `height` вместе с URL и текстовыми метаданными.
- Подготовка новой фотографии требует ImageMagick и ExifTool; точные команды и проверки находятся в storage runbook.
- Новый media host уже разрешен в CSP основного сайта; дополнительных внешних источников не появляется.
- Для другого раздела, формата или отдельного сервиса преобразований потребуется новое решение или расширение этого ADR.
