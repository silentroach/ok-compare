# ADR-001: Упоминания по слагу в Markdown

## Статус

Принят

## Дата

2026-05-13

## Контекст

На сайте уже есть профили людей в `/people/[slug]/` и Markdown-синтаксис `@slug` / `@slug:case`. Он решает важные задачи:

- автор пишет стабильный слаг, а не ручную ссылку на `/people/.../`;
- обработчик Markdown раскрывает упоминание в имя нужного падежа и ссылку на профиль;
- сборка проверяет неизвестные слаги и отсутствующие падежные формы;
- профили могут собирать упоминания и обратные ссылки из материалов сайта.

Этого достаточно, когда в тексте нужно показать имя человека: `@kschemelinin` или `@kschemelinin:gen`.

Но часть редакционных упоминаний естественнее писать не именем, а ролью или контекстным описанием:

```markdown
По словам [главного по электричеству](@kschemelinin), работы идут.
```

Такой формат лучше читается в тексте и не заставляет добавлять в `name_cases` одноразовые фразы, которые не являются формами имени.

Та же задача появится у будущих сущностей, например у карты мест: место может быть упомянуто каноническим названием или контекстной фразой вроде `[кафе у въезда](@green-dreams)`.

## Решение

Поддерживать два синтаксиса упоминаний по слагу в Markdown-обработке приложения.

### Каноническое упоминание

```markdown
@kschemelinin
@kschemelinin:gen
```

Использовать, когда в тексте нужно показать каноническое имя человека или название сущности.

Обработчик Markdown должен:

- найти слаг в реестре;
- выбрать `nom` без модификатора или явно указанный падеж;
- завершить сборку ошибкой, если слаг неизвестен или нужная форма не заполнена;
- заменить упоминание на ссылку с каноническим текстом;
- записать упоминание для обратных ссылок и машиночитаемых представлений.

### Упоминание с авторским текстом

```markdown
[главного по электричеству](@kschemelinin)
[кафе у въезда](@green-dreams)
```

Использовать, когда видимый текст должен остаться авторской фразой: роль, описание, короткий контекст или грамматическая конструкция, которую нельзя считать формой имени.

Обработчик Markdown должен:

- считать адрес ссылки вида `@slug` целью упоминания, а не обычной ссылкой;
- найти слаг в реестре;
- завершить сборку ошибкой, если слаг неизвестен;
- заменить адрес на канонический URL сущности;
- оставить видимый текст ровно таким, как написал автор;
- записать упоминание для обратных ссылок и машиночитаемых представлений.

Падежный модификатор внутри упоминания с авторским текстом не нужен: `[видимый текст](@slug:gen)` должен считаться ошибкой или не поддерживаться первой реализацией. Грамматика в этом варианте живет в видимом тексте, а не в реестре.

## Область действия

Первый реестр - люди из `people`. Будущие реестры, например `places`, должны использовать тот же принцип, если их Markdown-обработка поддерживает упоминания.

Если в одной Markdown-обработке одновременно доступны несколько реестров, одинаковые слаги между ними нужно запрещать на сборке. Синтаксис с пространством имен вроде `@place:green-dreams` не вводим до реального пересечения, потому что он делает обычные тексты шумнее.

## Закрытые спорные вопросы

### Два синтаксиса вместо одного

Принимаем оба варианта, но с разными ролями. Каноническое упоминание остается обычным вариантом для имен и названий. Упоминание с авторским текстом нужно только там, где видимый текст является ролью, описанием или частью фразы.

### Источник истины

Каноническое имя, URL и падежные формы остаются в данных сущности. Авторский текст в `[текст](@slug)` не становится новым именем сущности и не должен попадать обратно в `name_cases` или похожие поля.

### Падежи в ссылочном формате

Формат `[текст](@slug:case)` не принимаем. Если автор сам пишет видимый текст, он сам отвечает за грамматику этой фразы; обработчику достаточно проверить слаг и поставить правильную ссылку.

### Пересечения слагов

Короткий синтаксис важнее преждевременного пространства имен. Если несколько реестров подключены к одной Markdown-обработке, пересечение слагов должно быть ошибкой сборки. Пространство имен можно добавить отдельным ADR, если пересечения станут неизбежными.

### Люди и места

Решение принимается для общей модели упоминаний. Реализация может идти поэтапно: сначала люди, затем места. При добавлении мест не нужно заново выбирать синтаксис; нужно только подключить второй реестр и проверку пересечений.

## Рассмотренные альтернативы

### Оставить только `@slug` и `@slug:case`

Плюсы:

- самая строгая модель с единым источником истины;
- все видимые имена и падежи живут в данных сущности;
- проще валидировать и индексировать.

Минусы:

- плохо работает для ролей и описаний;
- провоцирует добавлять в `name_cases` не формы имени, а редакционные фразы;
- делает текст менее естественным там, где имя не лучший видимый текст ссылки.

Отклонено: этот вариант слишком жесткий для редакционных материалов.

### Перейти только на `[видимый текст](@slug)`

Плюсы:

- автор полностью контролирует видимый текст;
- не нужны падежные формы для грамматики конкретной фразы.

Минусы:

- ослабляет единый источник истины;
- легко получить разные написания одного имени в разных материалах;
- хуже подходит для массовой генерации машиночитаемых представлений, где важно раскрытое каноническое имя.

Отклонено: каноническое упоминание остается более безопасным вариантом по умолчанию.

### Разрешить ручные Markdown-ссылки на `/people/[slug]/`

Плюсы:

- ничего не нужно добавлять в Markdown-обработку;
- автор может написать любой текст ссылки.

Минусы:

- сборка не проверяет, что профиль существует;
- обратные ссылки и граф упоминаний теряют часть источников;
- URL-структура начинает протекать в контент.

Отклонено: ручные ссылки хуже, чем упоминание по слагу, как редакционный и машинный контракт.

### Сразу ввести синтаксис с пространством имен

Пример: `@people:kschemelinin`, `@place:green-dreams`.

Плюсы:

- нет неоднозначности между реестрами;
- проще расширять модель на новые типы сущностей.

Минусы:

- синтаксис заметно шумнее;
- текущие упоминания людей уже используют короткие слаги;
- реального пересечения пока нет.

Отклонено сейчас: пространство имен можно добавить позже, если появятся неизбежные пересечения.

## Последствия

- Существующие `@slug` и `@slug:case` остаются валидными и рекомендуются по умолчанию.
- Упоминание с авторским текстом становится запасным редакционным вариантом для ролей и описаний.
- Данные сущности остаются источником истины для канонического имени, URL и обратных ссылок, но не для каждого возможного видимого текста.
- Markdown-обработчик приложения должен обрабатывать адреса ссылок вида `@slug` до HTML-рендера и не трогать обычные внешние ссылки.
- Документация для авторов должна объяснять выбор: каноническое упоминание для имени, упоминание с авторским текстом для роли или контекстной фразы.
- Выгрузки для агентов должны учитывать оба источника упоминаний, чтобы обратные ссылки не зависели от выбранного синтаксиса.

## Заметка по реализации

На момент создания ADR код уже поддерживал канонические упоминания людей. На 2026-06-08 упоминания с авторским текстом тоже реализованы в общем app-level слое `apps/www/src/lib/mentions`: `[текст](@slug)` раскрывается в URL сущности, сохраняет авторский видимый текст и попадает в тот же набор упоминаний для обратных ссылок. Формат `[текст](@slug:case)` остается неподдерживаемым и должен падать на сборке.
