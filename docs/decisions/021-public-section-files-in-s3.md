# ADR-021: Публичные файлы разделов в S3

## Статус

Принят

Правила фотографий новостей позднее определены в [ADR-023](023-news-images-in-public-s3.md).

## Дата

2026-07-13

## Контекст

Разделам сайта нужны публичные PDF и другие скачиваемые документы. Хранить бинарные файлы в Git неудобно: они увеличивают репозиторий и его историю, а изменения между редакциями нельзя содержательно проверить как обычный текстовый diff. Ссылки только на внешние источники тоже недостаточны, потому что источник может заменить или удалить файл.

ADR-019 уже определяет публичный origin `https://media.kpshelkovo.online` перед бакетом `kps-public`, но не устанавливает, какие файлы следует хранить в S3, как связывать их с разделами сайта и как сохранять исторические редакции.

Сейчас объектное хранилище используется для документов базы знаний `/kb`, исходных PDF промо-раздела `/815/regulation` и фотографий новостей. Правила подготовки и публикации новостных изображений определены отдельно в ADR-023.

## Решение

- Хранить в публичном S3-совместимом бакете документы и скачиваемые вложения разделов, которые не должны находиться в Git. Markdown, метаданные, код, build assets и другие исходники проекта остаются в репозитории.
- Первый сегмент S3 key обозначает раздел-владелец файла. Для базы знаний используется префикс `kb/`, для промо-раздела `/815` - `815/`. Внутренняя структура ключа служит организации хранилища и не обязана повторять вложенные URL раздела.
- Если файл используют несколько разделов, его ключ сохраняет исходный раздел-владелец. Дополнительные ссылки не требуют копирования или переноса объекта.
- Публиковать ссылки только через стабильный origin `https://media.kpshelkovo.online/<key>`. Провайдерский URL бакета не является публичным контрактом сайта.
- Управлять объектами через S3 API endpoint `https://s3.regru.cloud`. Credentials хранятся вне репозитория и не должны попадать в документацию, команды, логи или ответы агента.
- Документы и скачиваемые вложения загружать без изменения их содержимого. Изображения можно готовить перед публикацией только по отдельному принятому решению для конкретного раздела; для фотографий новостей действует ADR-023.
- Для документов по умолчанию считать историю редакций важной. Каждую редакцию хранить под отдельным датированным ключом и не перезаписывать.
- Дата в ключе является датой снимка: самой ранней подтвержденной датой, когда проект видел эту конкретную редакцию. Для нового файла это дата получения по часовому поясу `Europe/Moscow`. Для переноса существующего материала можно использовать дату первого Git-коммита, подтверждающего эту редакцию. Если дату подтвердить нельзя, используется фактическая дата получения.
- Не определять дату снимка по локальному времени создания файла, S3 `Last-Modified` или автоматически по дате, напечатанной внутри документа.
- Недатированный изменяемый ключ использовать только по явному требованию, что история файла не нужна. Агент не должен самостоятельно относить документ к этой категории.
- Исходные PDF промо-раздела `/815/regulation` хранить под недатированными ключами `815/regulation/<filename>.pdf`: они поддерживают текущую промо-поверхность и не являются историческим архивом.
- В публичных данных и discovery-поверхностях ссылаться на внешний документ через S3 key и канонический media URL. Не публиковать внутренний путь файла в репозитории как часть контракта.
- Если объект по целевому ключу уже существует, сравнить его с переданным файлом. Одинаковый объект повторно не загружать; при различии остановиться и сообщить о конфликте.
- Не удалять опубликованные объекты без явного запроса и проверки известных ссылок. Опубликованный URL считается долгоживущим.
- Подробный порядок публикации и текущие параметры бакета хранить в `ops/storage/public-media.md`.

## Рассмотренные альтернативы

### Хранить бинарные файлы в Git

Отклонено: крупные и меняющиеся бинарные файлы раздувают историю репозитория, не дают полезного diff и связывают публикацию документа со сборкой сайта.

### Ссылаться только на внешний источник

Отклонено: источник может удалить файл, заменить его содержимое по прежнему URL или изменить домен. Это не сохраняет историческую редакцию, на которую опирается материал сайта.

### Публиковать URL провайдера S3

Отклонено ADR-019: такие URL привязывают публичный контракт сайта к провайдеру и обходят управляемые nginx кеш и защитные заголовки.

### Группировать все документы под общим префиксом `documents/`

Отклонено: верхнеуровневый префикс должен показывать раздел-владелец и назначение объекта. Внутри `kb/` можно использовать любую устойчивую структуру, не копируя маршруты базы знаний.

### Сохранять историю перезаписью одного ключа

Отклонено: публичный media-origin игнорирует query string, а текущие объекты не имеют S3 version ID. Исторические редакции должны иметь самостоятельные ключи и URL независимо от возможностей провайдера.

## Последствия

- Репозиторий хранит текстовое представление материалов и ссылки, но не публичные бинарные файлы.
- Раздел-владелец виден из ключа, при этом внутренняя структура хранилища может развиваться независимо от URL сайта.
- Публичные данные `/815/regulation` используют `pdf_key` и `pdf_url` вместо прежнего внутреннего `pdf_path`.
- Историческая редакция остается доступной после публикации новой, а актуальный материал сайта указывает на нужный неизменяемый URL.
- Публикация файла становится отдельной операцией с обязательной проверкой существующего объекта и результата через media-origin.
- Текущий инструмент не дает атомарной гарантии create-only, поэтому операции с одним key нельзя выполнять конкурентно, а содержимое нужно повторно проверять после загрузки.
- Неизменяемые редакции занимают дополнительное место в бакете, но не зависят от S3 versioning и не теряются при обычном обновлении материала.
- Существующие ключи сохраняются как опубликованные адреса и не требуют миграции.
- ADR-023 определяет преобразования, размеры и формат фотографий новостей. Изображения других разделов по-прежнему требуют отдельного решения.
