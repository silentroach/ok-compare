# ADR-012: Единый граф упоминаний сущностей в Markdown

## Статус

Принят

## Дата

2026-05-20

## Контекст

ADR-001 зафиксировал редакционный синтаксис упоминаний через `@slug`, `@slug:case` и `[видимый текст](@slug)`. ADR-003 зафиксировал, что такие доменные preprocessors живут в app-level Markdown-обертке `apps/www`, а не в общем пакете `@shelkovo/markdown`.

Сейчас реальный первый реестр - люди из раздела `people`. Упоминания людей уже должны работать в новостях, статусе и профилях людей. В будущем тот же редакционный контракт нужен для мест и, возможно, других сущностей сайта: автор должен писать стабильный `slug` в Markdown, а сборка должна раскрывать ссылку, проверять существование сущности и строить обратные ссылки.

Текущая реализация графа людей слишком связана с источниками. Модуль `people` строит обратные ссылки из новостей и статуса сам: знает заголовки статей, URL инцидентов, Markdown-URL, excerpt и ключи сортировки. Это делает граф людей владельцем деталей других разделов и усложняет подключение новых Markdown-поверхностей.

Нужно сохранить простоту редакционного синтаксиса и при этом подготовить границу, через которую любой раздел с редакционным Markdown body сможет публиковать упоминания людей, а позже - мест или других сущностей.

## Решение

Вводим единый app-level слой упоминаний сущностей для `apps/www`. Рабочее место для общего кода - `apps/www/src/lib/mentions`.

Этот слой отвечает за общий контракт упоминаний, но не становится публичным универсальным API сайта. Публичные поверхности остаются доменными: раздел `people` продолжает публиковать данные людей и их `backlinks`, будущий раздел мест будет публиковать данные мест и их `backlinks`.

Общий слой должен содержать нейтральные понятия:

- `EntityMentionType` - тип упоминаемой сущности, сначала `person`, позже `place` или другие реальные типы;
- `EntityMentionTarget` - цель упоминания: `type`, `slug`, `label`, необязательные `label_cases`, `html_url`, `markdown_url` и необязательный контекст для `title` ссылки;
- `SiteMentionRegistry` - объединенный реестр всех сущностей, доступных в одной Markdown-обработке;
- `EntityMentionSourceRef` - нормализованная ссылка на source unit, где упомянута сущность;
- `EntityMentionGraph` - индекс обратных ссылок по паре `type + slug`.

Для людей доменная модель остается человеко-ориентированной: профиль по-прежнему хранит `name` и `name_cases`. Адаптер людей мапит их во внутренние `label` и `label_cases`. Будущие места смогут мапить свое название и формы названия в тот же внутренний контракт.

## Markdown-поверхности

Упоминания сущностей работают только в редакционном Markdown body, который loader явно пропускает через app-level pipeline `preprocessSiteMarkdown` или `renderMarkdown`.

В область входят:

- body новости;
- body инцидента статуса;
- body профиля человека;
- будущие section-defined Markdown body units, если раздел явно подключает общий `SiteMentionRegistry`.

В область не входят заголовки, summary, подписи, URL, frontmatter-строки, сгенерированные Markdown-документы, `llms.txt` и любые другие plain-text поля, пока они явно не объявлены редакционным Markdown body. Это предотвращает ложные срабатывания в обычных строках и сохраняет границу ADR-003.

Дополнения к новостям не входят в новый контракт source units. Их не нужно учитывать при проектировании адаптеров графа.

## Реестр и короткие slug

Синтаксис остается коротким: `@slug`, `@slug:case`, `[видимый текст](@slug)`. `namespace` вроде `@person:slug` или `@place:slug` сейчас не вводится.

`SiteMentionRegistry` собирается до нормализации Markdown body из identity/frontmatter данных сущностей, а не из уже обработанных body. Это нужно, чтобы люди могли упоминать будущие места, будущие места могли упоминать людей, и loader не создавал циклическую зависимость между полными datasets.

Если несколько подключенных реестров содержат один и тот же short slug, сборка должна падать. Короткий синтаксис важнее преждевременного `namespace`, но неоднозначность между реестрами недопустима.

`@slug:case` использует нейтральные `label_cases`. Если указанная форма отсутствует, сборка должна падать. Это сохраняет текущее поведение людей и делает тот же контракт применимым к будущим местам.

## Самоупоминания

Самоупоминание сущности запрещено и должно падать на сборке, а не молча фильтроваться из графа.

Для Markdown body самой сущности loader передает в preprocessor `source_entity`, например `{ type: 'person', slug }`. Если в этом body найдено упоминание той же пары `type + slug`, preprocessor завершает сборку ошибкой рядом с исходным текстом.

Graph module может иметь защитную проверку на self-link, но основное место ошибки - нормализация Markdown body. Автор должен исправить исходный Markdown, а не получать тихо измененный набор обратных ссылок.

## Source refs и graph

Обратные ссылки строятся на уровне source unit, а не на уровне каждого occurrence в тексте.

Один source unit дает не больше одного `EntityMentionSourceRef` для одной target entity, даже если `@slug` встречается в body несколько раз. `mentions[]`, возвращаемый preprocessor, также остается уникальным списком target entities внутри одного Markdown body.

`EntityMentionSourceRef` строят модули источников или их соседние адаптеры, а не общий graph module. Источник владеет своими публичными ссылками, заголовками, excerpt, датой и ключом сортировки.

Примеры ответственности источников:

- новости знают, как превратить body статьи в ref с заголовком статьи, HTML URL, Markdown URL, excerpt, датой публикации и `sort_key`;
- статус знает, как превратить body инцидента в ref с маршрутом инцидента, Markdown URL, excerpt, датой и `sort_key`;
- people знает, как превратить body профиля в ref профиля и как передать `source_entity` для запрета self-link.

Общий graph module принимает уже готовые `EntityMentionSourceRef[]`, индексирует их по `target_type + target_slug`, группирует по source section и прикрепляет обратные ссылки к нужным доменным данным. Он не должен знать формы `NewsArticle`, `StatusIncident`, `PersonProfile` или будущих datasets.

Сортировка обратных ссылок внутри группы является частью публичного поведения:

- сначала `sort_key` по убыванию;
- затем `title` по `ru`;
- затем `source_id`;
- отсутствующий `sort_key` считается ниже источников с датой.

## Публичный контракт

На этом этапе не вводим публичный универсальный `/entities` API и не публикуем общий graph feed.

Причины:

- сейчас публично существует реальный потребитель графа людей;
- будущие места еще не имеют собственной доменной публичной поверхности;
- преждевременный общий API заставит стабилизировать поля, пока нет второго типа сущности и реальных сценариев чтения.

Раздел `people` продолжает публиковать `backlinks` в своей структуре, совместимой с текущей моделью людей. Внутренняя реализация может перейти на общий graph, но внешний people-контракт не обязан становиться generic.

## Рассмотренные альтернативы

### Оставить graph people-only

Плюсы:

- минимальное изменение текущего кода;
- не нужно вводить новые нейтральные типы;
- текущий публичный people-контракт уже работает.

Минусы:

- при появлении мест придется снова переносить ту же архитектурную границу;
- `people` продолжит знать детали новостей, статуса и будущих разделов;
- новые Markdown body будут подключаться через правки внутренностей людей, хотя проблема уже не people-only.

Отклонено: задача формулируется как единый редакционный контракт упоминаний сущностей, а люди - только первый реальный реестр.

### Сразу ввести публичный generic graph API

Плюсы:

- единая точка чтения всех связей;
- будущие сущности сразу попадут в общий формат;
- проще строить внешние обходчики полного графа.

Минусы:

- пока нет второго реального типа сущности и сценариев чтения общего графа;
- публичный generic API будет трудно менять после публикации;
- доменные разделы все равно должны показывать свои backlinks в своем языке и UI.

Отклонено сейчас: общий graph нужен как внутренний слой сборки, а публичные контракты должны оставаться доменными до появления реального второго потребителя.

### Ввести `namespace` в Markdown-синтаксис сейчас

Пример: `@person:kschemelinin`, `@place:green-dreams`.

Плюсы:

- нет неоднозначности между типами сущностей;
- легче читать источник для машинной обработки;
- можно разрешить одинаковые slug в разных реестрах.

Минусы:

- редакционный синтаксис становится шумнее;
- текущие материалы и ADR-001 уже используют короткий `@slug`;
- реального конфликта slug пока нет;
- преждевременный `namespace` ухудшит обычное написание текстов ради гипотетического пересечения.

Отклонено сейчас: сохраняем короткий `@slug`, а пересечения slug между подключенными реестрами считаем ошибкой сборки. `namespace` можно ввести отдельным ADR, если конфликт станет неизбежным.

### Собирать occurrence-level graph

Плюсы:

- можно знать точное число упоминаний;
- можно показывать несколько фрагментов одного source unit;
- полезно для будущей аналитики плотности упоминаний.

Минусы:

- текущий пользовательский вопрос звучит как "где упоминается", а не "сколько раз встретился token";
- нужны позиции, контекст вокруг каждого occurrence и более сложная дедупликация;
- UI и публичные feeds стали бы шумнее.

Отклонено: backlinks остаются source-level. Occurrence-level данные можно добавить позже отдельным контрактом, если появится реальный сценарий.

### Искать упоминания во всех текстовых полях

Плюсы:

- меньше риска забыть упоминание в коротком поле;
- можно писать `@slug` в title или summary.

Минусы:

- обычные строки начнут вести себя как Markdown без явного объявления;
- возрастет риск ложных срабатываний в URL, служебных текстах и сгенерированных документах;
- нарушится граница ADR-003, где доменные preprocessors применяются к body markdown.

Отклонено: mention-enabled surface - только явно обработанный редакционный Markdown body.

### Молча фильтровать self-link

Плюсы:

- проще сохранить текущий подход, где self backlink не попадает в профиль;
- авторский Markdown не ломает сборку.

Минусы:

- исходный текст содержит некорректную ссылку на самого себя, но build это скрывает;
- будущие сущности получат разные неявные правила;
- ошибка обнаружится только при анализе графа, а не рядом с исходным body.

Отклонено: self-link является ошибкой редакционного контракта и должен падать на build.

## Последствия

- Новый раздел с редакционным Markdown body должен подключать общий `SiteMentionRegistry`, а не отдельный people-only preprocessor.
- Данные mention registry должны собираться из identity/frontmatter слоя до нормализации body, чтобы разные типы сущностей могли ссылаться друг на друга без циклов datasets.
- `people/mentions.ts` является адаптером людей поверх общего `lib/mentions`, а не владельцем всего синтаксиса упоминаний.
- `news`, `status` и `people` отдают source refs через свои адаптеры, потому что только источник знает собственные title, URL, excerpt, дату и `sort_key`.
- Общий graph module тестируется на группировку, дедупликацию source-level refs, сортировку и ошибки self-link, не используя реальные формы `NewsArticle` и `StatusIncident`.
- Тесты источников проверяют свои adapters: какие refs публикует новость, инцидент или профиль.
- Публичные people surfaces должны сохранить поведение `backlinks.news`, `backlinks.status` и `backlinks.people`, даже если внутренняя реализация переедет на общий graph.
- При появлении мест нужно добавить registry adapter мест, source refs для их Markdown body и доменную публичную поверхность мест, а не менять синтаксис Markdown заново.

## Заметка по реализации

На 2026-06-08 общий слой `apps/www/src/lib/mentions` реализован. Он содержит registry, нормализацию `@slug`/`@slug:case`/`[текст](@slug)`, source-level refs, граф обратных ссылок, дедупликацию, сортировку и проверку self-link. `apps/www/src/lib/people/mentions.ts` стал адаптером профиля человека в generic mention target, а `news`, `status` и `people` публикуют source refs через свои соседние adapters. Публичный people-контракт остается доменным; `EntityMentionType` пока содержит только `person`, потому что второго реального типа сущности еще нет.
