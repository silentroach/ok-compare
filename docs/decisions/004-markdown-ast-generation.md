# ADR-004: Генерация Markdown через AST

## Статус

Принят

## Дата

2026-05-14

## Контекст

На сайте есть несколько публичных текстовых поверхностей, которые отдаются как Markdown или фактически используют Markdown-разметку:

- сопровождающие Markdown-страницы новостей, статуса, людей, регламента и Compare;
- индексные Markdown-файлы для машинного обхода;
- `llms.txt` и `llms-full.txt`.

Эти поверхности являются публичным контрактом для терминалов, агентов, поисковых роботов и прямых ссылок. Для них важны стабильная структура, корректное экранирование, единый стиль списков, ссылок, кода, таблиц и frontmatter.

Ручная сборка Markdown плохо масштабируется как архитектурный подход.

Проблемы ручной сборки:

- экранирование Markdown-символов остается ответственностью каждого генератора;
- frontmatter и тело документа легко получают разные правила сериализации;
- вложенные списки, ссылки, код, таблицы и будущие GFM-конструкции превращаются в ручную работу со строками;
- единый стиль вывода приходится поддерживать договоренностью, а не общим кодом;
- тесты начинают проверять случайную раскладку строк вместо структуры документа.

При этом в репозитории уже есть общий пакет `@shelkovo/markdown`. По ADR-003 он отвечает за универсальную Markdown-инфраструктуру, а слой приложения подключает доменные препроцессоры для рендера тела Markdown в HTML. Значит генерация Markdown тоже должна жить в общем пакете как универсальная функция, а не в `apps/www`.

Важный нюанс: генераторы не только создают Markdown из структурных данных. Они также вставляют уже подготовленные Markdown-фрагменты из редакционного контента, например `article.body`, `incident.body` и `profile.body`. Новый API должен уметь парсить такие фрагменты в mdast и вклеивать их в итоговый документ, а не превращать их в экранированный обычный текст.

## Решение

Весь Markdown и Markdown-совместимый публичный текст сайта должен генерироваться через AST, а не ручной сборкой строк.

В качестве AST используем mdast. Это совпадает с уже выбранной экосистемой unified/remark и с текущим опытом команды.

`@shelkovo/markdown` получает API для сборки, парсинга фрагментов и сериализации mdast-документов. Генераторы в `apps/www` должны строить mdast-деревья или mdast-фрагменты и отдавать их в пакетный сериализатор.

Базовый стек:

- `mdast-util-to-markdown` для сериализации mdast в Markdown;
- `mdast-util-from-markdown` для парсинга существующих Markdown-фрагментов в mdast;
- `mdast-util-gfm` для GFM-сериализации и парсинга, включая таблицы, списки задач, автоматические ссылки, сноски и зачеркнутый текст;
- `mdast-util-frontmatter` для YAML-узла frontmatter в mdast;
- `yaml` для сериализации frontmatter-объектов в YAML.

`remark-stringify` не выбираем как основной API генерации. Он остается допустимой оберткой для unified-цепочек, но для сборки документов нам нужен прямой контроль над mdast-деревом и настройками сериализатора. `remark-stringify` внутри опирается на тот же `mdast-util-to-markdown`, поэтому прямой пакет проще для API сборки.

## Пакетный API

`@shelkovo/markdown` должен экспортировать небольшой универсальный слой для генерации:

- `serializeMarkdownDocument(document, options?)` - принимает mdast `Root` или описание документа и возвращает строку Markdown с финальным переводом строки;
- `parseMarkdownFragment(markdown, options?)` - парсит исходный Markdown в `readonly RootContent[]`, чтобы вставлять редакционные фрагменты тела документа;
- `createMarkdownDocument({ frontmatter, children })` - создает mdast `Root`, добавляя YAML frontmatter первым узлом, если он передан;
- `md` или аналогичный набор тонких функций для частых узлов: heading, paragraph, text, link, list, listItem, inlineCode, code, blockquote, thematicBreak;
- типы публичного контракта, которые не заставляют приложение импортировать внутренности сериализатора напрямую.

API не должен становиться отдельным языком разметки. Предпочтение - тонкие функции поверх стандартных типов mdast и возможность передать обычные узлы mdast без оберток.

## Frontmatter

Frontmatter генерируется как узел mdast `{ type: 'yaml', value }` и сериализуется через `mdast-util-frontmatter`.

Значение `value` строится через современный YAML-сериализатор, сейчас кандидат - пакет `yaml`.

Пакет задает единые настройки для стабильного вывода:

- блочные коллекции;
- отступ 2 пробела;
- без YAML-директив внутри значения узла;
- без переноса коротких скалярных значений, если это не нужно явно.

Значения не должны оборачиваться в кавычки всегда по умолчанию. YAML-сериализатор должен ставить кавычки только там, где они нужны для сохранения смысла значения: даты, URL, строки с двоеточиями, значения, похожие на boolean, и другие неоднозначные скалярные значения.

Места вызова по-прежнему решают, какие поля включать. Пакет не должен сам удалять доменные необязательные поля вроде `areas` для записей, которые относятся ко всему поселку. Он может предоставить общую функцию для удаления `undefined`, но смысловые правила остаются в разделах.

`null` не используется. Отсутствующие значения передаются как `undefined` или не включаются в объект frontmatter.

## Markdown-Фрагменты

Для вставки уже написанного Markdown используем `parseMarkdownFragment`.

Примеры источников:

- `article.body` и тело дополнений к новостям;
- `incident.body`;
- `profile.body`;
- будущие кураторские фрагменты в данных регламента.

Фрагменты парсятся с тем же GFM-набором, что и сериализатор. Это дает нормальное экранирование и единый стиль вывода, но сохраняет Markdown-семантику авторского текста.

Если фрагмент содержит frontmatter, это должно быть отдельным явно описанным случаем. По умолчанию фрагмент тела не должен приносить свой frontmatter в середину сгенерированного документа.

## Стиль Markdown

Сериализатор в пакете задает единый стиль генерируемого Markdown:

- ATX-заголовки;
- ненумерованные списки через `-`;
- нумерованные списки через `1.`;
- огражденные блоки кода через обратные кавычки;
- GFM-таблицы через `mdast-util-gfm`;
- финальный перевод строки в каждом документе;
- экранирование небезопасных Markdown-символов силами сериализатора, а не ручными регулярными выражениями в приложении.

Точные настройки сериализатора считаются частью контракта пакета. Если меняем стиль вывода, меняем его централизованно и обновляем тесты.

## Агентские Тексты

`llms.txt` и `llms-full.txt` тоже попадают под это решение.

Хотя расширение у них `.txt`, по структуре это Markdown-совместимые агентские документы: заголовки, списки, ссылки, встроенный код и разделы. Они должны собираться тем же AST-подходом, чтобы не заводить второй способ генерации тех же структур.

## Связь с ADR-003

ADR-003 остается в силе.

Этот ADR регулирует генерацию Markdown: структурные данные плюс mdast плюс сериализатор в строку Markdown.

ADR-003 регулирует рендер Markdown: исходный Markdown плюс препроцессоры приложения плюс пакетная цепочка рендера в HTML.

`@shelkovo/markdown` держит оба универсальных слоя, но упоминания, обратные ссылки и доменная валидация остаются в `apps/www/src/lib/markdown/render.ts` и загрузчиках данных.

## Рассмотренные альтернативы

### Оставить строковые генераторы и добавить общие вспомогательные функции

Плюсы:

- минимальная миграция;
- меньше новых зависимостей;
- снимки тестов почти не меняются.

Минусы:

- проблема ручного Markdown не решается;
- экранирование ссылок, заголовков и списков остается ответственностью каждого места вызова;
- frontmatter остается отдельным ручным форматом;
- новые сложные конструкции снова будут собираться строками.

Отклонено: это косметическое улучшение текущей проблемы, а не архитектурное решение.

### Использовать только `remark-stringify`

Плюсы:

- привычная unified-цепочка;
- хорошо сочетается с текущим подходом `@shelkovo/markdown` к рендеру;
- плагины подключаются так же, как в цепочке рендера.

Минусы:

- для ручной сборки AST все равно нужны узлы mdast;
- `remark-stringify` скрывает низкоуровневый сериализатор, который нам как раз нужен как пакетный примитив;
- сложнее сделать маленький сфокусированный API для сборки документов и парсинга фрагментов.

Отклонено как основной API. Можно использовать позже, если появится отдельная цепочка преобразований, где unified-подход оправдан.

### Использовать отдельный Markdown DSL не из unified/mdast

Плюсы:

- потенциально более компактный API;
- можно быстро подобрать библиотеку с цепочным сборщиком.

Минусы:

- новый Markdown AST будет жить рядом с уже используемым unified/mdast;
- нужно отдельно решать GFM, frontmatter, парсинг фрагментов и совместимость с HTML-рендером;
- меньше шансов переиспользовать существующий пакет и знания команды.

Отклонено: mdast уже соответствует текущему стеку и предпочтениям команды.

### Генерировать Markdown через шаблоны

Плюсы:

- визуально проще для длинных статичных документов;
- меньше AST-кода для простых страниц.

Минусы:

- шаблоны все равно требуют ручного экранирования;
- вложенные списки, ссылки и необязательные блоки быстро становятся условными строками;
- frontmatter и фрагменты остаются отдельными проблемами.

Отклонено для генерируемого Markdown. Шаблоны допустимы только для почти статичных документов без динамических Markdown-сущностей.

## Последствия

- В `@shelkovo/markdown` появятся новые зависимости и публичный API генерации.
- `apps/www/src/lib/markdown/frontmatter.ts` должен исчезнуть или стать временным тонким алиасом на пакетную функцию frontmatter до полной миграции.
- Генераторы разделов мигрируются по одному, со снимками или точечными тестами на публичные URL, frontmatter и важные Markdown-контракты.
- Вывод Markdown может измениться в безопасных местах: экранирование, отступы списков, формат ссылок, пустые строки между блоками.
- Существующий вид Markdown не сохраняем специально. После миграции правильным считается вывод пакетного сериализатора, если AST-семантика и публичные ссылки сохраняются.
- Новые генераторы Markdown после принятия ADR должны использовать пакетный API, а не ручную сборку строк.
- Отдельное ограничение импортов или архитектурный тест сейчас не нужен. Решение фиксируется ADR и соглашением в коде; проверку можно добавить позже, если появится реальная регрессия.
