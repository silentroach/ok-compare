# ADR-004: Lighthouse-профиль без аналитики

## Статус

Принят

## Дата

2026-05-15

## Контекст

`kpshelkovo.online` использует Yandex Metrika на боевом сайте. Lighthouse Best Practices стабильно снижает оценку из-за сторонних cookie и срабатываний `inspector-issues` из `mc.yandex.ru` / `mc.yandex.com`. Это внешний сервисный эффект, а не регрессия собственного кода сайта.

Полностью убирать Метрику с боевого сайта нельзя без отдельного продуктового решения: она нужна для рабочей аналитики. Одновременно плановый Lighthouse CI должен оставаться полезным сигналом по качеству кода и страниц, а не регулярно шуметь из-за принятой сторонней интеграции.

## Решение

Боевая сборка и Lighthouse-проверка боевого сайта сохраняют Yandex Metrika включенной.

Плановый/статический Lighthouse CI строит сайт с `LIGHTHOUSE_DISABLE_ANALYTICS=true`. Этот флаг на этапе сборки отключает только вставку Metrika в статический Lighthouse-артефакт и не меняет поведение боевого сайта.

Для цели `production` в Lighthouse CI порог `categories:best-practices` снижен до `0.75`, потому что реальная боевая страница намеренно включает Metrika и получает принятые срабатывания по сторонним cookie. Цель `static` сохраняет `0.9` как основной барьер для регрессий в собственном коде.

Сводка Lighthouse должна явно показывать состояние аналитики и, если отчет все же содержит замечания Yandex Metrika в `third-party-cookies` или `inspector-issues`, объяснять, что это принятое ограничение боевой интеграции.

## Рассмотренные альтернативы

### Убрать Yandex Metrika с боевого сайта

Плюсы:

- исчезает срабатывание по сторонним cookie;
- CSP можно сузить для источников Metrika.

Минусы:

- теряется рабочая аналитика;
- это продуктовое решение, а не техническая правка Lighthouse.

Отклонено: задача не должна менять рабочую аналитику.

### Оставить Метрику включенной во всех Lighthouse-прогонах

Плюсы:

- Lighthouse всегда измеряет ровно боевой HTML;
- не нужен отдельный профиль на этапе сборки.

Минусы:

- плановый CI продолжит шуметь по принятому стороннему сервису;
- предупреждение Best Practices будет плохо отличаться от настоящей регрессии в собственном коде.

Отклонено для планового/статического CI: основная автоматическая проверка должна контролировать качество собственного кода.

### Полностью отключить проверки Best Practices

Плюсы:

- исчезают ложные предупреждения из-за Metrika.

Минусы:

- теряются полезные сигналы по другим регрессиям Best Practices;
- будущие регрессии станут менее заметны.

Отклонено: цель `static` должна продолжать держать высокий порог.

## Последствия

- `LIGHTHOUSE_DISABLE_ANALYTICS` должен быть учтен в `turbo.json` как `env`-зависимость результата сборки.
- Плановый Lighthouse CI проверяет статический сайт без Metrika.
- Ручная цель `production` в Lighthouse остается способом увидеть реальное поведение боевого сайта с Metrika.
- CSP боевого сайта сохраняет источники Metrika, пока рабочая аналитика включена.
