# ADR-020: Отдельное Astro-приложение для media error pages

## Статус

Принят

## Дата

2026-07-11

## Контекст

Media-origin проксирует объекты S3 и не является разделом основного сайта. При этом отсутствующие объекты должны возвращать локальную фирменную страницу с корректным статусом `404` вместо XML-ответа S3 `AccessDenied`.

Размещение страницы внутри `apps/www` связывает media-origin с глобальными настройками основного сайта: prefetch runtime, client router, аналитикой, sitemap и общим deploy-root. Ручной HTML в `public` обходит Astro и дублирует UI, логотип и шрифты.

## Решение

- Создать отдельное статическое Astro-приложение `apps/media` с production artifact `dist/media`.
- Приложение не включает prefetch, client router, аналитику и клиентские скрипты.
- Единственный route `src/pages/404.astro` штатно собирается в `404.html`; стили встраиваются в документ, а оптимизированные изображения и шрифты попадают в `/_media/`.
- Общий визуальный блок обеих страниц 404 живет в `@shelkovo/ui/NotFoundView.astro`. Канонический логотип хранится в `packages/ui/src/assets/logo.png`.
- Nginx использует отдельный static root `/var/www/media-kpshelkovo-online`, отдает из него только internal `404.html` и зарезервированный asset-префикс `/_media/`. Остальные пути проксируются в S3.
- Turbo собирает `apps/www` и `apps/media` как независимые workspace-приложения. CI синхронизирует `dist/www` и `dist/media` в разные каталоги VPS.

## Рассмотренные альтернативы

### Оставить страницу в `apps/www`

Отклонено: media 404 получает ненужные глобальные возможности основного приложения или требует специальных исключений для prefetch и deploy-путей.

### Хранить готовый HTML в `apps/www/public`

Отклонено: файл не проходит через Astro, а общий визуальный блок и брендовые assets приходится поддерживать отдельно.

### Встроить логотип и шрифты в HTML

Отклонено: каждый ответ на отсутствующий объект заметно увеличивается, даже если запрос пришел из `img`, `video` или ссылки на документ и браузер не покажет HTML.

## Последствия

- Media 404 имеет независимый build и deploy boundary без специальных исключений в основном приложении.
- Nginx-конфиг использует простой отдельный root и не зависит от hashed assets `apps/www`.
- Изменения общего not-found UI автоматически затрагивают оба приложения через workspace-зависимость.
- Локальный `pnpm dev` запускает основной сайт на `:4321` и media app на `:4322`.
- `deploy-nginx-site` автоматически создает `/var/www/media-kpshelkovo-online` с правами пользователя деплоя до синхронизации artifact.
