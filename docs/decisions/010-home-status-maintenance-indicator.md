# ADR-010: Плановые работы в индикаторе статуса на главной

## Статус

Принят

## Дата

2026-05-18

## Контекст

Главная страница `kpshelkovo.online` показывает рядом со ссылкой на `/status/` маленький цветной индикатор состояния поселковых сервисов. До этого индикатор различал только два состояния:

- красный, если на момент сборки есть активный `incident`;
- зеленый, если активных `incident` нет.

В разделе `/status/` уже есть третий тип фактического состояния: активные плановые работы `maintenance` должны давать `amber`, если по сервису нет активного `incident`. Это состояние важно показывать и на главной, потому что пользователь видит краткий статус до перехода в подробный журнал.

Сайт собирается как статическое Astro-приложение. HTML главной может быть собран до начала плановых работ, а открыт пользователем уже во время этих работ. Только build-time расчет в таком случае оставит индикатор зеленым до следующей сборки.

Плановых работ обычно немного, поэтому отдельный клиентский запрос ради нескольких временных окон избыточен. При этом автообновление открытой вкладки без перезагрузки не требуется.

## Решение

На главной странице используем три состояния индикатора: `red`, `amber`, `green`.

Приоритет состояний фиксируется так:

- `red`: есть активный `incident` на момент сборки;
- `amber`: активных `incident` нет, но есть активный `maintenance` на момент сборки;
- `green`: нет активных `incident` и `maintenance` на момент сборки.

Дополнительно в HTML главной встраивается инертный JSON со всеми будущими относительно времени сборки окнами плановых работ:

- только записи `kind: maintenance`;
- только записи с `started_at` позже времени сборки страницы;
- только записи с явным `ended_at`;
- формат данных минимальный: `start` и `end` как timestamp в миллисекундах.

Клиентский код после загрузки страницы один раз читает встроенный JSON и текущее состояние индикатора:

- если состояние уже `red`, клиентский код ничего не меняет;
- если текущее время попадает хотя бы в одно окно `start <= now < end`, состояние меняется на `amber`;
- `aria-label` ссылки `/status/` обновляется вместе с визуальным состоянием;
- таймеры для будущего переключения состояния не ставятся.

Встроенные данные должны оставаться данными, а не исполняемым inline-кодом. Для этого используется блок вида `script type="application/json"` или эквивалентный инертный HTML-контейнер. Исполняемая логика должна оставаться в обработанном клиентском скрипте проекта, согласованно с ADR-005.

## Рассмотренные альтернативы

### Считать оранжевый только на этапе сборки

Плюсы:

- самая простая реализация;
- не нужен клиентский код для индикатора.

Минусы:

- статический HTML может устареть к моменту открытия страницы;
- плановые работы, начавшиеся после сборки, не будут отражены до следующего билда.

Отклонено: главная должна корректно показать уже начавшиеся плановые работы без пересборки.

### Загружать плановые работы AJAX-запросом

Плюсы:

- можно получать самые свежие данные;
- HTML главной остается меньше.

Минусы:

- добавляется сетевой запрос ради нескольких временных окон;
- появляется дополнительная точка отказа и задержка до корректного цвета;
- усложняется кеширование и тестирование простого индикатора.

Отклонено: объем данных мал, а нужная точность достигается встроенным JSON.

### Автообновлять индикатор через таймеры

Плюсы:

- вкладка, открытая до начала работ, сама переключится в нужный момент;
- можно автоматически вернуть зеленый цвет после окончания окна.

Минусы:

- больше клиентской логики и edge cases вокруг сна вкладки, переходов Astro и нескольких окон;
- для текущего пользовательского сценария достаточно проверки при загрузке.

Отклонено: автообновление без перезагрузки явно не требуется.

### Позволить плановым работам переопределять красный статус

Плюсы:

- индикатор может показывать, что отключение было ожидаемым.

Минусы:

- активный `incident` является более критичным состоянием;
- оранжевый цвет может скрыть реальную проблему, если одновременно идут плановые работы и есть авария.

Отклонено: приоритет должен быть `red > amber > green`.

## Последствия

- Главная получает тот же смысл `amber`, что уже используется в статусных сводках сервисов.
- Будущая реализация должна сериализовать только минимальные временные окна, без title, service и body, если они не нужны для индикатора.
- Клиентская логика должна быть одноразовой: загрузка страницы и Astro client navigation, без polling и без `setTimeout` на будущие окна.
- Тесты должны покрывать приоритет `red`, попадание зеленого состояния в окно плановых работ и границы `start <= now < end`.
- Если в будущем потребуется показывать пользователю детали работ прямо на главной, нужно отдельно пересмотреть формат встроенных данных.
