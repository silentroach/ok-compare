# ADR-005: CSP без `unsafe-inline` для исполняемых скриптов

## Статус

Принят

## Дата

2026-05-15

## Контекст

Lighthouse `csp-xss` указывает на два класса риска в боевой CSP: `script-src 'unsafe-inline'` разрешает любые встроенные скрипты и обработчики событий, а список разрешенных сторонних хостов для Yandex Metrika и Yandex Maps расширяет поверхность доверия.

Сайт статический, поэтому nonce на уровне запроса неудобен: nginx не генерирует уникальные nonce для каждого HTML-ответа, а nonce, заданный на этапе сборки для всех страниц, не дает нужного свойства одноразовости. CSP-хэш для всего встроенного кода на уровне страницы тоже хрупок: Astro, JSON-LD и страничные скрипты меняют итоговый HTML при сборке.

MDN описывает отказ от встроенных скриптов/стилей как один из главных выигрышей CSP и рекомендует nonce или хэш, если встроенный код неизбежен: https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy/script-src#unsafe_inline_script.

Astro поддерживает обработанные клиентские скрипты из `src/`, которые собираются как обычные браузерные модули: https://docs.astro.build/en/guides/client-side-scripts/#script-processing.

## Решение

Исполняемый встроенный код сайта вынесен в `apps/www/src/scripts/site-runtime.ts` и подключается через обработанный Astro `<script src="../scripts/site-runtime.ts"></script>` из `BaseLayout.astro`.

В `apps/www/astro.config.ts` выставлен `vite.build.assetsInlineLimit: 0`, чтобы Astro не встраивал маленькие обработанные скрипты вроде общего клиентского модуля и обработчиков на уровне страницы. После этого в HTML остаются только сгенерированные Astro фрагменты загрузки Svelte-островов; они разрешены через SHA-256-хэши в nginx `script-src`.

Этот клиентский модуль отвечает за:

- индикатор навигации при переходах Astro;
- отложенную загрузку Yandex Metrika;
- отправку `hit` Metrika после клиентских переходов;
- скрытие статического резервного списка `/815/compare/` после готовности `SettlementsExplorer`.

Боевая CSP-политика удаляет `script-src 'unsafe-inline'`, добавляет хэши для фрагментов загрузки Astro-островов и задает `script-src-attr 'none'`, чтобы встроенные обработчики событий были явно запрещены. `style-src 'unsafe-inline'` остается, потому что переходы Astro и отдельные HTML-атрибуты `style` пока требуют встроенных стилей; это не разрешает исполняемый JavaScript.

Список разрешенных хостов для `mc.yandex.ru`, `mc.yandex.com`, `api-maps.yandex.ru` и `yastatic.net` остается как документированное исключение: Metrika и Yandex Maps являются принятыми боевыми интеграциями. Их удаление требует отдельного продуктового решения.

Lighthouse при `script-src` на основе хэшей предлагает добавить `'unsafe-inline'` для обратной совместимости со старыми браузерами, где хэши не поддерживаются. Мы не добавляем этот резервный вариант: для поддерживаемых современных браузеров хэши работают, а возвращение `'unsafe-inline'` ухудшает читаемость политики и может снова попасть в аудиты безопасности как широкое разрешение.

## Рассмотренные альтернативы

### CSP на основе nonce

Плюсы:

- точечно разрешает встроенные скрипты;
- совместимо с динамически сгенерированным HTML.

Минусы:

- для статической сборки нужен слой, который генерирует уникальный nonce на каждый ответ и меняет HTML;
- усложняет nginx и деплой без практической необходимости после выноса встроенного кода.

Отклонено: текущий сайт статический, а исполняемый встроенный код можно убрать проще.

### CSP на основе хэшей для всех встроенных блоков

Плюсы:

- работает со статическим HTML;
- не требует nonce на этапе выполнения.

Минусы:

- хэш чувствителен к пробелам и итоговой минификации;
- JSON-LD и вывод, сгенерированный Astro, усложняют поддержку;
- каждое изменение встроенного кода требует синхронного обновления CSP.

Отклонено для написанного в проекте кода и кода на уровне страниц: дешевле и надежнее держать исполняемый код во внешнем собранном ресурсе. Принято только для небольших сгенерированных фрагментов загрузки Astro-островов, которые остаются в HTML после отключения встраивания обработанных скриптов.

### Убрать списки разрешенных хостов Yandex Metrika и Yandex Maps

Плюсы:

- CSP стал бы строже;
- Lighthouse перестал бы жаловаться на списки разрешенных хостов.

Минусы:

- ломает рабочую аналитику;
- ломает интерактивные карты в разделе сравнения;
- это продуктовое решение, а не безопасная инфраструктурная правка.

Отклонено в этой задаче: внешние интеграции остаются принятым риском.

## Последствия

- Новые браузерные скрипты в `BaseLayout.astro` и страницах не должны использовать `is:inline` для исполняемого кода.
- Если нужен новый исполняемый встроенный скрипт, сначала рассмотреть `src/scripts/*` или компонентный обработанный `<script>` без `is:inline`.
- Если хэши встроенных скриптов, сгенерированных Astro, меняются после обновления Astro/Svelte или клиентских директив, нужно синхронно обновить nginx `script-src` и заново проверить сборку, приближенную к боевой.
- CSP `script-src` не должен возвращать `'unsafe-inline'` без новой ADR или явной проверки безопасности.
- `style-src 'unsafe-inline'` остается отдельным будущим направлением усиления безопасности.
