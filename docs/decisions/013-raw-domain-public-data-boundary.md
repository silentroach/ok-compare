# ADR-013: Граница между внешними DTO, доменной моделью и публичными DTO

## Статус

Принят

## Дата

2026-05-20

## Контекст

В `apps/www` данные приходят из YAML-файлов и frontmatter. В исходниках данных используется `snake_case`: это удобно для редакционных файлов и уже принято в текущем контенте. Дальше эти данные валидируются через Zod-схемы и часто сразу становятся внутренними TypeScript-структурами через `z.infer`.

Из-за этого внешний формат протекает внутрь приложения. Внутренний код работает с полями вроде `short_name`, `address_text`, `normalized_per_sotka_month`, `started_iso`, `name_cases`. То же относится к строковым значениям enum-полей: внешние значения вроде `rub_per_sotka`, `partial_asphalt` и `checkpoint_only` становятся внутренними доменными значениями без явного решения.

Это создает несколько проблем:

- внутренние интерфейсы выглядят как копия YAML, а не как доменная модель приложения;
- Zod-схема одновременно становится валидатором, источником доменных типов и местом неявных преобразований;
- невозможно ясно понять, какие поля являются внешним контрактом, какие являются внутренним языком приложения, а какие публикуются наружу;
- публичные JSON/agent surfaces из ADR-011 могут случайно получить внутреннюю форму данных или устаревший raw-формат;
- mapper-слой отсутствует, поэтому мы упускаем возможность упростить доменную модель при загрузке данных.

Нужно жестко разделить данные снаружи и данные внутри. Снаружи могут быть YAML, frontmatter, внешние API, публичные JSON и другие опубликованные поверхности. Внутри должен быть собственный доменный язык TypeScript-кода.

## Решение

Вводим три отдельных слоя данных.

### Raw DTO

`Raw DTO` описывает входной внешний формат: YAML, frontmatter или ответ внешнего сервиса. Его поля и строковые значения соответствуют источнику данных, даже если там `snake_case`.

Для `Raw DTO` используем Zod-схемы. Они отвечают за валидацию внешнего формата, обязательные default-значения и минимальные нормализации на уровне источника, нужные именно для чтения этого источника. Они не являются доменной моделью приложения.

Имена:

- `RawSettlementSchema`, `RawNewsArticleSchema`, `RawStatusIncidentSchema`;
- `RawSettlement`, `RawNewsArticle`, `RawStatusIncident`;
- тип `Raw*` можно получать из Zod через `z.output<typeof Raw*Schema>`.

### Domain model

`Domain model` описывает внутренний язык приложения. Это вручную написанные TypeScript interfaces и type aliases с `readonly`-полями, `camelCase`-именами и JSDoc-комментариями для неочевидных полей.

Доменные данные после mapper-а считаются иммутабельным снимком данных на этапе сборки. По возможности они должны быть целиком readonly: все свойства объявляются `readonly`, коллекции описываются как `readonly T[]` или `ReadonlyMap`, вложенные объекты имеют собственные readonly interfaces вместо изменяемых inline-структур. Это TypeScript-контракт, а не требование делать runtime `Object.freeze()`. Если нужно изменить данные, код должен построить новый доменный объект или отдельное представление, а не мутировать результат mapper-а.

Доменная модель не импортирует Zod и не выводится через `z.infer`. Если поле во внешнем источнике называется плохо или слишком технически, mapper может дать ему более точное доменное имя. Например, доменная модель не обязана сохранять форму `normalizedPerSotkaMonth`, если точнее назвать поле по смыслу.

Строковые значения enum-полей тоже считаются частью границы. Внутренние union types должны использовать наш TypeScript-формат и доменный смысл. Многочастные внешние значения вроде `rub_per_sotka`, `partial_asphalt` и `checkpoint_only` не должны протекать внутрь без явного маппинга. Если однословное значение совпадает с внешним, оно все равно должно проходить через явный mapper, а не попадать внутрь автоматически.

### Mapper

`Mapper` является единственным местом перевода из `Raw DTO` в `Domain model`.

Mapper получает на вход тип, выведенный из Zod-схемы, и возвращает вручную написанный domain interface. В mapper-е можно:

- переводить `snake_case` в `camelCase`;
- переводить внешние enum/string literal values во внутренние значения;
- схлопывать неудобные внешние структуры в более простую доменную форму;
- вычислять производные поля, если они являются частью внутреннего представления на этапе сборки;
- выбрасывать ошибки инвариантов, которые относятся к сборке доменной модели, а не к синтаксису исходного файла.

### Public DTO

`Public DTO` описывает данные, которые сайт публикует наружу: JSON, discovery, схемы в стиле OpenAPI, agent-facing feeds и другие стабильные public surfaces.

Публичный DTO не должен быть ни raw-форматом, ни доменной моделью напрямую. Он получает отдельный тип и отдельный adapter из доменной модели. Это сохраняет свободу менять внутренние интерфейсы без случайного изменения public contract.

Для новых публичных JSON и agent-facing DTO используем `camelCase`, если внешний стандарт явно не требует другого формата. Существующие опубликованные поверхности считаются контрактом: их изменение должно быть отдельным осознанным шагом с обновлением схем, discovery, llms-документов и тестов.

## Организация файлов

Схемы, интерфейсы и mapper-ы должны лежать в разных файлах.

Рекомендуемый минимальный набор для доменной области:

- `schema.ts` или `raw-schema.ts` - Zod-схемы `Raw DTO`;
- `types.ts` - вручную написанные domain interfaces и domain type aliases;
- `mapper.ts` - `Raw DTO -> Domain model`;
- `public.ts` или `public-dto.ts` - опциональный adapter `Domain model -> Public DTO`.

Если область большая, файлы лучше складывать в папку рядом с доменным модулем, например `settlement/schema.ts`, `settlement/types.ts`, `settlement/mapper.ts`.

Правило зависимостей:

- `types.ts` не импортирует Zod;
- `schema.ts` не импортирует domain interfaces;
- `mapper.ts` импортирует raw-типы и domain interfaces;
- внутренний app/lib/component-код импортирует domain interfaces и loaders, а не raw-типы;
- public adapters импортируют domain interfaces и возвращают явные public DTO;
- domain interfaces должны быть readonly по всей вложенности, насколько это практически возможно без искусственных generic wrappers.

## Разрешенные места для внешнего формата

`snake_case` и внешние enum values допустимы только там, где код явно работает с внешней границей:

- raw Zod-схемы;
- mapper-ы из raw-формата в домен;
- adapters из домена в legacy public DTO, если такой контракт пока сохраняется;
- тестовые fixtures, которые имитируют исходные YAML/frontmatter или legacy public output;
- тексты документации, где описывается внешний формат.

Во внутренней доменной модели, вычислениях, UI-компонентах и build-time агрегаторах `snake_case` не должен быть обычным способом обращения к данным.

## Рассмотренные альтернативы

### Оставить `z.infer` как доменную модель

Плюсы:

- меньше файлов;
- текущий код продолжает работать без миграции;
- схема и тип всегда синхронизированы технически.

Минусы:

- внешний формат остается внутренним контрактом;
- невозможно документировать доменную модель отдельно от Zod-схемы;
- enum/string literal values протекают внутрь без явного решения;
- любое улучшение YAML-формата становится риском для внутреннего кода.

Отклонено: это сохраняет текущую архитектурную проблему.

### Делать `transform()` в Zod-схеме сразу в доменную модель

Плюсы:

- Zod может вернуть уже `camelCase`-данные;
- меньше явных функций mapper-а;
- loader выглядит короче.

Минусы:

- схема смешивает валидацию внешнего формата и построение домена;
- сложные преобразования становятся скрыты внутри цепочки валидаторов;
- вручную написанные interfaces и JSDoc хуже отделяются от схемы;
- труднее тестировать маппинг отдельно от валидации.

Отклонено как основной подход. Zod может делать нормализацию на уровне источника, но доменный перевод должен жить в mapper-е.

### Использовать общий recursive `snakeToCamel()`

Плюсы:

- быстро убирает `snake_case` из объектов;
- требует мало ручного кода;
- легко применить ко всем YAML сразу.

Минусы:

- меняет написание, но не уточняет смысл;
- не решает enum/string literal values;
- не позволяет упростить неудобные raw-структуры;
- ошибки маппинга становятся runtime-сюрпризами, а не явным контрактом.

Отклонено: нам нужен доменный mapper, а не механическая смена регистра.

### Использовать domain model как public DTO

Плюсы:

- меньше типов и adapters;
- проще отдать данные наружу;
- public output всегда соответствует текущему внутреннему состоянию.

Минусы:

- публичный контракт начинает тормозить внутренний рефакторинг;
- внутренние вычисленные поля могут случайно стать опубликованными;
- невозможно сохранить legacy public contract без загрязнения домена;
- ADR-011 требует относиться к public surfaces как к отдельному контракту.

Отклонено: public DTO должен быть отдельной внешней границей.

### Сделать миграцию одним большим изменением

Плюсы:

- быстрее прийти к чистому состоянию;
- меньше временных адаптеров;
- проще сформулировать финальный grep-инвариант.

Минусы:

- большой радиус регрессий;
- сложно параллелить работу агентов;
- тяжело отделить изменение доменных типов от изменения public output;
- UI, loaders, schemas и agent-facing поверхности будут меняться одновременно.

Отклонено: миграция должна идти маленькими вертикальными задачами.

## Последствия

- Для каждого нового YAML/frontmatter-источника нужны raw Zod-схема, domain types и mapper.
- Domain interfaces пишутся руками и документируются JSDoc там, где смысл поля не очевиден из имени.
- Domain interfaces являются readonly snapshot-контрактом: `readonly` properties, readonly collections и readonly nested interfaces.
- `z.infer` и `z.output` допустимы для raw-типа, но не являются способом объявить domain model.
- Loaders должны маппить данные в домен как можно ближе к `getCollection()` или другому внешнему входу.
- Public JSON и agent-facing feeds должны строиться через отдельные public adapters.
- Existing public surfaces нельзя случайно менять во время внутренней миграции: если public shape меняется, это отдельный task с обновлением схем, discovery, llms и тестов.
- Enum/string literal mappings должны быть явными и тестируемыми.
- Для текущих доменов финальная миграция оставила `snake_case` только в разрешенных внешних зонах.

## Заметка по реализации

На 2026-06-08 текущие домены `news`, `status`, `people`, `meetings` и Compare разведены по этой схеме: raw Zod-схемы и raw-типы живут на внешней границе, mapper-ы переводят данные в вручную описанные readonly domain interfaces, а публичные JSON/agent-facing payloads собираются через отдельные public DTO adapters. В UI-страницах и компонентах `apps/www` не используются legacy `snake_case` поля вроде `started_at`, `name_cases`, `source_urls` или compare enum values; они остаются в raw/public DTO, mapper-ах, fixtures и документации внешнего формата.
