# ADR-003: Markdown-модель данных мест

## Статус

Принят

## Дата

2026-05-13

## Контекст

[ADR-002](002-places-static-editorial-map.md) фиксирует раздел `/places/` как предсобранную редакционную карту мест. Для этого раздела нужна база, которую можно постепенно наполнять без сложной админки: сначала редакцией и активными жителями, позже при необходимости более формальным процессом.

У места есть две разные части:

- структурные данные для карты, фильтров, контактов, ссылок и фото;
- обычное человеческое описание: почему место полезно, кому подойдет и что проверить перед поездкой.

Чистый YAML хорошо подходит для структурных данных, но неудобен для редакционного описания. Чистый Markdown без схемы удобен для текста, но не дает надежной карты, фильтров и проверок на сборке.

Текущее Astro-приложение уже хранит контентные данные в `apps/www/src/data` и подключает их через `defineCollection` с загрузчиком `glob`. Для `places` нужно следовать этому же стилю, а не заводить отдельное хранилище.

## Решение

Добавить коллекцию контента `places`, основанную на Markdown-файлах с frontmatter в `apps/www/src/data/places`.

Описание места живет в теле Markdown-документа. Frontmatter хранит только структурные поля, которые нужны карте, фильтрам, ссылкам, контактам, источникам и фото.

Поддерживать две формы хранения:

```text
apps/www/src/data/places/green-dreams.md
```

Для мест без локальных вложений.

```text
apps/www/src/data/places/green-dreams/index.md
apps/www/src/data/places/green-dreams/cover.jpg
apps/www/src/data/places/green-dreams/interior.jpg
```

Для мест с фото или другими локальными файлами.

Канонический слаг места равен имени `.md` файла или папки. Переопределение `slug` в первой версии не вводим: пока нет опубликованных адресов и миграций, отдельное поле создаст лишний источник расхождений. Если появится реальная потребность в сохранении старого адреса, это нужно оформить отдельным изменением схемы.

## Минимальная схема

Обязательные поля первой версии:

- `title` - название места;
- `type` - один тип из контролируемого словаря;
- `scenarios` - один или несколько сценариев из контролируемого словаря;
- `location.lat` - широта;
- `location.lng` - долгота.

Желательные, но необязательные поля:

- `summary` - короткая подпись для карточек списка;
- тело Markdown-документа - основное редакционное описание;
- `location.address` - человекочитаемый адрес;
- `location.map_url` - ссылка на Яндекс Карты или другой внешний маршрутный источник;
- `contacts` - сайт, телефон, Telegram, Instagram, VK или другие контакты;
- `links` - источники, внешние карточки, отзывы;
- `photos` - локальные фото с `alt` и опциональной подписью;
- `notes` - практические заметки: парковка, дети, доставка, средний чек, особенности визита.

Если `summary` отсутствует, карточка списка может брать выдержку из тела Markdown-документа. Если описание пустое, место все равно может быть валидным, если у него есть обязательные поля для карты и фильтров.

## Контролируемые словари

Первый словарь типов:

- `cafe`;
- `restaurant`;
- `shop`;
- `clinic`;
- `doctor`;
- `pharmacy`;
- `kids`;
- `sport`;
- `poi`;
- `service`.

Первый словарь сценариев:

- `with-guests`;
- `with-kids`;
- `quick-food`;
- `on-the-way-home`;
- `health`;
- `walk`;
- `useful-nearby`.

Словари должны быть перечислениями, а не свободными строками. Новые значения можно добавлять, но только осознанно: иначе фильтры быстро расползутся и перестанут помогать выбору.

## Пример

```markdown
---
title: Green Dreams
type: cafe
scenarios:
  - with-guests
  - with-kids
  - quick-food
summary: Кафе рядом с поселком, которое можно рассматривать для встречи с гостями или семейного обеда.
location:
  address: Московская область, Ступино, примерный адрес
  lat: 55.000000
  lng: 37.000000
  map_url: https://yandex.ru/maps/...
contacts:
  - type: website
    label: Сайт
    value: https://example.com
  - type: phone
    label: Телефон
    value: '+7 000 000-00-00'
links:
  - title: Карточка на Яндекс Картах
    url: https://yandex.ru/maps/...
    type: maps
  - title: Отзывы
    url: https://example.com/reviews
    type: reviews
photos:
  - src: ./cover.jpg
    alt: Зал кафе Green Dreams
    caption: Основной зал кафе.
notes:
  parking: Парковка рядом со входом.
  kids: Есть детские стульчики.
---

Green Dreams можно рассматривать как место для спокойной встречи с гостями или семейного обеда рядом с КП Шелково.

Перед поездкой лучше проверить актуальное время работы во внешней карточке места.
```

## Упоминания мест

[ADR-001](001-markdown-slug-mentions.md) уже фиксирует общую модель упоминаний по слагу и отдельно упоминает будущий реестр `places`. Для первой версии раздела `/places/` упоминания мест не обязательны.

Если места позже подключаются к Markdown-упоминаниям, они должны следовать ADR-001:

- каноническое упоминание `@green-dreams` раскрывается в название места и ссылку;
- авторское упоминание `[кафе у въезда](@green-dreams)` сохраняет видимый текст и ставит ссылку на место;
- формы для падежей хранятся в данных места, например в `title_cases`;
- если явно указан падеж, но нужной формы нет, сборка должна падать с понятной ошибкой;
- если одна Markdown-обработка видит и `people`, и `places`, одинаковые слаги между реестрами должны быть ошибкой сборки.

Пространство имен вроде `@place:green-dreams` не вводим заранее. Оно станет отдельным решением только при реальном и неизбежном пересечении слагов.

## Рассмотренные альтернативы

### Чистые YAML-файлы

Плюсы:

- проще парсить как структурные данные;
- меньше вариантов разметки;
- удобно для коротких записей без описания.

Минусы:

- редакционное описание приходится хранить в YAML-строке;
- авторам сложнее писать несколько абзацев, списки и ссылки;
- карточки мест становятся похожи на сухой каталог.

Отклонено: для этого раздела важен человеческий редакционный слой, а не только координаты и контакты.

### TypeScript или JSON как источник данных

Плюсы:

- строгая типизация;
- легко импортировать в код;
- удобно для разработчиков.

Минусы:

- выше порог входа для редакторов и жителей;
- сложнее добавлять обычное описание;
- больше риск превратить наполнение в разработческую задачу.

Отклонено: первая версия должна проверять удобство наполнения, а не требовать кода для каждого места.

### Внешняя CMS или таблица

Плюсы:

- удобнее массовый ввод для нетехнических авторов;
- можно добавить роли, формы и проверку;
- проще собирать данные без изменений в репозитории.

Минусы:

- новая внешняя зависимость;
- отдельная модель доступа и синхронизации;
- лишняя сложность до проверки полезности раздела.

Отклонено для первой версии: сначала достаточно файлового источника в репозитории.

### Обязательные богатые карточки

Плюсы:

- каждая карточка сразу выглядит полной;
- больше данных для пользователя;
- меньше пустых отдельных страниц мест.

Минусы:

- высокий порог добавления нового места;
- авторы будут откладывать публикацию из-за отсутствия фото, телефона или заметок;
- часть данных быстро устаревает.

Отклонено: обязательными должны быть только поля, без которых не работают карта и фильтры.

## Последствия

- Добавить место можно одним Markdown-файлом.
- Более богатое место может хранить локальные фото рядом с `index.md`.
- Схема должна валидировать обязательные координаты, типы и сценарии на сборке.
- Слаг места остается стабильным и читаемым, потому что идет от имени файла или папки.
- Редакторы могут писать описание обычным Markdown, а сайт может типографировать его через существующую Markdown-обработку приложения.
- При подключении упоминаний мест нужно переиспользовать ADR-001, а не изобретать новый синтаксис.
