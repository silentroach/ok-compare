# ADR-014: Архив встреч как публичная доказательная поверхность

## Статус

Принят

## Дата

2026-05-26

## Контекст

На сайте нужен способ сохранять и переиспользовать суть поселковых встреч, созвонов, эфиров и письменных ответов так, чтобы жителям не приходилось смотреть полную запись ради базового понимания, а внешние читатели и AI-агенты могли проверить источник, найти расшифровку и сослаться на конкретный фрагмент.

Новости уже являются главным редакционным входом для обычных читателей. Но новость плохо подходит как долговременный архив первоисточника: она рассказывает событие, а не хранит весь контекст встречи, протокол, запись, расшифровку, документы и устойчивые якоря. Если складывать встречи только в новости, со временем появится выбор между перегруженной новостной страницей и потерей проверяемости.

В `apps/www` уже есть несколько решений, которые задают рамку для такого раздела:

- ADR-008 требует генерировать публичный Markdown через AST;
- ADR-009 регулирует согласование Markdown через `Accept` там, где это используется;
- ADR-011 вводит реестр публичных поверхностей;
- ADR-012 задает единый слой упоминаний сущностей в Markdown;
- ADR-013 разделяет исходные DTO, доменную модель и публичные DTO.

Архив встреч должен вписаться в эти решения, а не становиться отдельной медиа-платформой или вторым новостным разделом.

## Решение

Вводим публичный, но не продвигаемый через главное меню раздел встреч. Его роль - архив проверяемых первоисточников и редакционно обработанных итогов, на которые можно ссылаться из новостей, статусов, документов, корневых AI-поверхностей и внешних материалов.

Раздел публикует следующие поверхности:

- `/meetings/` - список встреч для людей, прямых ссылок и обхода AI-агентами;
- `/meetings/[date]/[slug]/` - HTML-страница конкретной встречи;
- `/meetings/[date]/[slug]/index.md` - Markdown-версия конкретной встречи;
- `/meetings/data/meetings.json` - структурированная публичная лента встреч;
- `/meetings/llms.txt` - краткое описание раздела для AI;
- `/meetings/llms-full.txt` - подробное описание форматов, URL, ограничений и рекомендуемого порядка чтения.

`/meetings/` не добавляем в главное меню и не добавляем в sitemap. Страницы конкретных встреч добавляем в sitemap, потому что они являются публичными источниками, на которые можно ссылаться из новостей, поисковых систем, документов и AI-поверхностей.

Дата в URL пишется в ISO-формате `YYYY-MM-DD` и стоит перед слагом: `/meetings/2026-05-26/example-meeting/`. Полная публичная идентичность встречи - пара `date + slug`; один и тот же слаг может повториться в разные даты. После публикации дата в URL считается стабильной частью публичного адреса. Если позже нужно исправить отображаемую дату уже опубликованной встречи, это должно делаться без тихой поломки старого URL или с явным перенаправлением.

HTML остается каноническим представлением для людей. Markdown, JSON и `llms*.txt` являются публичными поверхностями для терминальных клиентов, поисковых роботов и AI-агентов. Генерируемые Markdown- и `llms*.txt`-документы должны строиться через общий AST-подход из ADR-008. Новые публичные поверхности раздела должны быть описаны в реестре публичных поверхностей из ADR-011.

## Модель страницы

Страница встречи строится как документ с доказательной базой, а не как видеоплеер.

Главный принцип: сначала смысл, потом доказательства. Житель должен быстро понять, что обсуждали, что решили, что осталось открытым и почему это важно. Запись, расшифровка и исходные ссылки идут ниже по иерархии как материал для проверки, цитирования, чтения AI-агентами и внешнего контекста.

Страница состоит из секций, которые показываются только при наличии данных:

- главное: краткое описание, дата, формат, участники или организаторы, ключевые выводы, решения, открытые вопросы, последствия для жителей и связанные ссылки;
- протокол: повестка, решения, договоренности, следующие действия, вопросы жителей, ответы, статусы вопросов, спорные места, таймкоды и источники;
- запись и расшифровка: видео, аудио, ссылка на запись, полная расшифровка, спикеры, таймкоды, исходный файл или внешний источник.

Пустые секции и заглушки не показываем. Отсутствие видео, расшифровки или протокола не является ошибкой. Ошибка - показывать встречу так, будто уровень доказательности выше фактического набора источников.

Протокол не равен расшифровке. Протокол - редакционно обработанный слой смысла: решения, вопросы, договоренности, спорные места и статусы. Расшифровка - доказательная база, которая может быть сырой, длинной и неточной.

## Данные

Встреча валидна при минимальном наборе данных:

- `title`;
- `date`;
- `summary`;
- `slug`.

Остальные поля необязательны и добавляются по мере наличия источников:

- `format`;
- `source_url`;
- `video_url`;
- `video_embed_url`;
- `transcript`;
- `highlights`;
- `agenda`;
- `decisions`;
- `action_items`;
- `questions`;
- `participants`;
- `documents`.

Отдельные поля `audio_url`, `related_news` и `timestamps` в первой версии не вводим. Аудиозапись, если она появится без видео, можно дать как `source_url` или пересмотреть модель записи отдельным изменением. Связи с новостями должны строиться через упоминания встреч в редакционном Markdown, а не через ручной список `related_news` внутри встречи. Таймкоды относятся к протоколу или расшифровке и должны жить там, где находится сам фрагмент, а не отдельным верхнеуровневым списком.

Исходный редакционный формат может использовать `snake_case`, как остальные YAML/frontmatter-источники. Внутри приложения встречи должны следовать ADR-013: исходный DTO валидируется отдельно, доменная модель имеет `camelCase` и readonly-поля, публичный JSON строится отдельным адаптером и не отдает исходный формат напрямую.

Полную расшифровку храним отдельным сопутствующим файлом рядом со встречей, а не большим полем в frontmatter основной встречи. Базовое направление для расшифровки - YAML с сегментами, спикерами и таймкодами, но точная схема является контрактом реализации первой версии, а не частью этого ADR. Если схема расшифровки станет самостоятельным публичным или межредакционным контрактом, ее нужно зафиксировать отдельной спецификацией или ADR.

## Связи и якоря

Ссылки из новостей и других материалов должны оставаться простыми и устойчивыми:

- на встречу целиком: `/meetings/[date]/[slug]/`;
- на протокол: `/meetings/[date]/[slug]/#protocol`;
- на запись и расшифровку: `/meetings/[date]/[slug]/#recording`;
- на конкретный фрагмент: `/meetings/[date]/[slug]/#t-00-12-34` или другой стабильный якорь таймкода.

Встречи подключаем к общему слою упоминаний из ADR-012. Для ссылки на встречу целиком в редакционном Markdown используем `@meeting-id` или `[видимый текст](@meeting-id)`, где `meeting-id` - уникальный slug встречи в реестре упоминаний. В первой версии он выводится из публичной идентичности встречи как `YYYY-MM-DD-slug`; отдельное поле `mention_id` не вводим.

Упоминание встречи ведет на HTML-страницу встречи целиком. Если нужна ссылка на протокол, запись или конкретный таймкод, используем обычную Markdown-ссылку на URL с якорем. Это сохраняет простоту mention-слоя: он связывает сущности, а не отдельные фрагменты внутри сущности.

## Видео и CSP

Видео можно встраивать через iframe только для разрешенных провайдеров и только после проверки CSP в `ops/nginx/*`. Раздел не должен принимать произвольный `video_embed_url` как доверенный iframe без разрешающего списка и проверки политики безопасности.

На первом этапе не строим общий видеохостинг и не добавляем сторонние SDK для встраивания скриптов. Сайт хранит смысл, структуру, ссылки, расшифровку и устойчивые якоря. Видео остается внешним источником или iframe-вставкой, если это безопасно и совместимо с CSP.

Точный список iframe-провайдеров не фиксируется этим ADR. Практическое правило для первой версии: начинать с первого реального провайдера, который нужен для опубликованной встречи, а не добавлять несколько провайдеров заранее.

## AI-доступ

Раздел должен быть удобен для AI через публичные документы, а не через отдельный чат-бот.

AI-поверхности должны давать агенту возможность ответить не только на вопрос "что сказали на встрече", но и на вопросы "какое решение приняли", "какие вопросы остались открытыми" и "где это проверить". Поэтому для AI важны не только сегменты расшифровки, но и редакционные поля:

- `summary`;
- `decisions`;
- `actionItems`;
- `openQuestions`;
- `sourceUrl`;
- якоря таймкодов;
- тело в Markdown или сегменты расшифровки.

Публичный JSON должен быть публичным DTO, а не исходным YAML и не внутренней доменной моделью напрямую. Если нужны схемы или записи обнаружения для новой ленты, они должны поддерживать ту же границу.

## Рассмотренные альтернативы

### Оставить встречи внутри новостей

Плюсы:

- не нужен новый раздел;
- новости уже являются привычным входом для читателей;
- меньше маршрутов и публичных поверхностей.

Минусы:

- новость начинает смешивать рассказ события и архив источника;
- расшифровка, запись, протокол и документы перегружают редакционную страницу;
- устойчивые ссылки на фрагменты и обход AI-агентами становятся случайной частью новости;
- одну встречу сложнее переиспользовать из нескольких материалов.

Отклонено: новости остаются входом и контекстом, но встреча должна иметь собственную страницу-источник.

### Сделать встречи видимым разделом главного меню

Плюсы:

- раздел легче найти обычному посетителю;
- больше шансов, что архив заметят без ссылки из новости;
- проще объяснить структуру сайта.

Минусы:

- встречи начинают конкурировать с новостями;
- архив может выглядеть как еще одна лента, хотя его задача другая;
- пустой или редкий архив в меню создает завышенное ожидание регулярности.

Отклонено для первой версии. Раздел публичный и ссылочный, но основным пользовательским входом остаются новости и прямые ссылки.

### Требовать видео или расшифровку для публикации

Плюсы:

- каждая встреча имеет сильную доказательную базу;
- структура страниц более однородна;
- меньше риска опубликовать слабый источник.

Минусы:

- полезные краткие итоги будут ждать идеальной расшифровки или записи;
- часть встреч и письменных ответов нельзя будет сохранить;
- редакционный процесс станет тяжелее и медленнее.

Отклонено: модель с необязательными полями важнее. Страница должна честно показывать доступные источники, а не блокировать публикацию без полного набора материалов.

### Называть расшифровку протоколом

Плюсы:

- меньше сущностей в интерфейсе и данных;
- можно быстро публиковать расшифровку как основной материал.

Минусы:

- сырая расшифровка не выделяет решения, статусы и договоренности;
- ошибки распознавания или устной речи начинают выглядеть как официальный итог;
- AI и жители хуже отличают факт произнесенной фразы от редакционного вывода.

Отклонено: расшифровка и протокол должны быть разными слоями.

### Сначала сделать AI-чат-бот по встречам

Плюсы:

- можно дать удобный диалоговый вход;
- часть пользователей не будет читать длинные документы;
- выглядит как быстрая AI-функция.

Минусы:

- без качественных публичных данных чат-бот будет отвечать по слабой базе;
- сложнее проверять источники и цитировать устойчивые URL;
- добавляются вопросы приватности, качества ответов и поддержки модели.

Отклонено для первого этапа. Сначала строим качественные публичные HTML, Markdown, JSON и `llms*.txt`-поверхности.

### Оставить связи с новостями только обычными Markdown-ссылками

Плюсы:

- не нужно расширять слой упоминаний;
- любая ссылка на якорь сразу работает обычным Markdown;
- меньше требований к уникальности slug встреч.

Минусы:

- сборка не проверяет, что ссылка на встречу существует;
- сложнее строить обратные ссылки от встречи к новостям;
- связь между новостью и встречей остается обычным URL, а не доменной связью;
- AI-поверхности хуже видят, какие материалы ссылаются на встречу.

Отклонено. Обычные Markdown-ссылки оставляем для якорей и внешних источников, но ссылку на встречу целиком лучше делать через общий mention-слой.

## Последствия

- Для встреч нужен новый доменный модуль данных с исходной схемой, типами доменной модели, преобразователем и адаптером публичного DTO по ADR-013.
- Для раздела нужны HTML-страницы, Markdown-версии, JSON-лента, `llms.txt`, `llms-full.txt` и фрагмент реестра публичных поверхностей по ADR-011.
- Markdown и `llms*.txt` раздела должны генерироваться через AST API из `@shelkovo/markdown`, а не ручной сборкой строк.
- Интерфейс должен покрывать разные уровни полноты данных: только итоги, итоги плюс видео, расшифровка без видео, полный протокол, полный набор источников.
- Тесты должны проверять не случайный текст страницы, а стабильные URL, наличие ключевых публичных поверхностей, правило sitemap для встреч, отсутствие пустых секций и корректность публичного DTO.
- При добавлении iframe-видео нужно синхронно проверить CSP в `ops/nginx/*`.
- При публикации расшифровки и спорных утверждений нужен редакционный контроль персональных данных, точности цитирования и ссылок на исходники.
- Новости продолжают быть главным входом для жителей и могут ссылаться на встречу через `@meeting-id`, а на ее якоря - обычными Markdown-ссылками.
- Если позже появится поиск по расшифровке, автоматическое разделение по говорящим или синхронизация видео и текста, это должно быть отдельным решением поверх этого ADR.

## Не делаем в первой версии

- Не добавляем раздел в главное меню.
- Не строим видеохостинг.
- Не требуем видео для публикации встречи.
- Не требуем расшифровку для публикации встречи.
- Не делаем поиск по расшифровке.
- Не делаем автоматическую генерацию расшифровки.
- Не делаем автоматическое разделение по говорящим.
- Не делаем плеер с синхронизацией текста и видео.
- Не делаем личный кабинет или сбор вопросов на сайте.
- Не делаем отдельные страницы вопросов жителей.
- Не вводим отдельные статусы доказательности.
- Не вводим ручное поле `related_news` в данных встречи.
- Не делаем AI-ответы внутри сайта.
- Не делаем глобальный календарь встреч.
