# ADR-003: Слоистый Markdown-рендер

## Статус

Принят

## Дата

2026-05-14

## Контекст

В репозитории есть два разных уровня Markdown-обработки:

- `@shelkovo/markdown` - общий пакет для Markdown-рендера и типографики;
- `apps/www/src/lib/markdown/render.ts` - обертка приложения корневого сайта.

Общий пакет уже решает универсальные задачи: GFM, безопасный HTML-вывод без raw HTML, типографику текстовых узлов, helpers для excerpt и форматирование коротких динамических HTML/text-фрагментов.

Сайт `kpshelkovo.online` решает дополнительные доменные задачи, которые не являются общей Markdown-функциональностью:

- упоминания доменных сущностей через `@slug`, `@slug:case` и `[видимый текст](@slug)`;
- проверку неизвестных сущностей и отсутствующих падежных форм на этапе сборки;
- сбор обратных ссылок и будущих связей между сущностями, специфичных для сайта;
- потенциальные preprocessors для будущих разделов сайта.

Если страницы или разделы импортируют `render` из `@shelkovo/markdown` напрямую, они получают корректный универсальный HTML, но могут случайно потерять доменную обработку сайта. Это особенно опасно для body markdown: ошибка выглядит как обычный рендер, но ломает редакционный контракт упоминаний, обратных ссылок и проверок на этапе сборки.

Одновременно нельзя переносить логику `people`, `news`, `status` и routes в общий пакет: тогда `@shelkovo/markdown` перестанет быть переиспользуемым универсальным пакетом и начнет зависеть от структуры конкретного Astro-приложения.

## Решение

Зафиксировать слоистую модель Markdown-рендера.

`@shelkovo/markdown` остается низкоуровневым универсальным пакетом. Он отвечает за безопасный Markdown-to-HTML pipeline, типографику и общие helpers, но не знает про `people`, `news`, `status`, routes сайта, обратные ссылки или доменные ошибки этапа сборки.

Типографические правила относятся к `@shelkovo/markdown`. Если нужно изменить типографику Markdown или коротких динамических HTML/text-фрагментов, это изменение должно попадать в пакет, а не в обертку приложения. Слой приложения решает, где применять типограф, но не держит собственный набор типографических правил.

`apps/www/src/lib/markdown/render.ts` является единственной точкой рендера body markdown внутри `apps/www`. Эта обертка вызывает пакетный `render` и подключает специфичные для сайта preprocessors через свои options.

Минимальный контракт:

- body markdown в `apps/www` рендерится через `@/lib/markdown/render`;
- прямой `render` из `@shelkovo/markdown` внутри кода приложения запрещен, кроме самой обертки и низкоуровневых тестов или конфигов, где явно не рендерится body markdown;
- `formatDynamicHtml`, `extractFirstMarkdownText` и `rehypeTypograf` можно импортировать напрямую из `@shelkovo/markdown`, потому что это не полноценный рендер body markdown;
- новые preprocessors, специфичные для сайта, добавляются в обертку приложения и ее options, а не в общий пакет;
- если будущему разделу нужно особое Markdown-поведение, он расширяет обертку/options, а не создает параллельный pipeline.

## Область действия

Решение относится к body markdown внутри `apps/www`: новости, incidents раздела `status`, профили людей и будущие разделы сайта с Markdown-контентом.

Решение не запрещает использовать универсальные helpers из `@shelkovo/markdown` для коротких строк, извлечения excerpt, Astro markdown config и низкоуровневых тестов пакета.

Решение не требует переносить весь Markdown-контент сайта в один формат данных. Оно фиксирует только границу: полноценный рендер body markdown должен проходить через обертку приложения.

## Закрытые спорные вопросы

### Где живут упоминания сущностей

Упоминания сущностей остаются на уровне приложения. Сейчас первый реестр - люди из `people`, но та же модель может использоваться для будущих мест, организаций или других сущностей сайта. Такие упоминания завязаны на локальные реестры, URL сущностей, падежные формы, ошибки этапа сборки и обратные ссылки. Для общего пакета это доменная логика, а не Markdown-примитив.

### Где живут типографические правила

Типографика остается в общем пакете. Это не доменная логика `apps/www`, а часть общего контракта Markdown-рендера и helpers для коротких динамических HTML/text-фрагментов. Обертка приложения может выбирать, какой контент прогонять через рендер или helper, но не должна заводить параллельные правила типографики.

### Одна обертка вместо нескольких pipeline

Новые разделы расширяют `@/lib/markdown/render`, а не заводят собственный независимый pipeline. Иначе сайт быстро получит несколько почти одинаковых Markdown-обработчиков с разным поведением упоминаний, типографики и HTML-безопасности.

### Прямые helper-импорты

Разрешаем прямые импорты helpers, которые не являются pipeline рендера body markdown. `formatDynamicHtml` нужен для коротких уже готовых HTML/text-фрагментов, `extractFirstMarkdownText` - для excerpt, `rehypeTypograf` - для конфигурации внешних Markdown-пайплайнов вроде Astro config.

### Проверка границы

Границу фиксируем ADR, `apps/www/AGENTS.md` и README пакета. Отдельный import-boundary test сейчас не обязателен: он даст ранний сигнал о прямом импорте `render`, но будет строковым архитектурным тестом с собственной хрупкостью и неполным покрытием всех вариантов импорта.

Если появится реальная регрессия или мест рендера body markdown станет заметно больше, можно добавить guard-тест или lint-правило. Это будет проверка уже принятого решения, а не новое архитектурное решение.

## Рассмотренные альтернативы

### Перенести упоминания сущностей в `@shelkovo/markdown`

Плюсы:

- любой вызов пакетного `render` автоматически получает упоминания;
- меньше риска забыть обертку приложения;
- один pipeline физически находится в пакете.

Минусы:

- общий пакет начинает знать про людей, будущие места, URL сайта и локальные контракты этапа сборки;
- переиспользование пакета в других приложениях становится сложнее;
- будущие доменные preprocessors будут тянуть за собой еще больше зависимостей от приложения.

Отклонено: упоминания и обратные ссылки являются доменной логикой сайта, а не общей Markdown-функцией.

### Разрешить страницам самим выбирать пакетный `render` или обертку приложения

Плюсы:

- меньше формальных ограничений;
- простые страницы могут рендериться напрямую через общий пакет;
- не нужен import-boundary тест.

Минусы:

- новое место рендера легко теряет упоминания сущностей и проверки на этапе сборки;
- поведение Markdown становится зависимым от внимательности автора страницы;
- доменные ошибки становятся тихими регрессиями в HTML.

Отклонено: body markdown должен иметь один безопасный вход в app-слое.

### Создать отдельные обертки для `news`, `status` и `people`

Плюсы:

- каждый раздел может явно описывать свои Markdown-особенности;
- локальная настройка кажется проще на старте.

Минусы:

- быстро появятся параллельные pipelines с частично разными правилами;
- общие доменные preprocessors придется дублировать или синхронизировать;
- трудно гарантировать, что общие упоминания и обратные ссылки работают одинаково во всех разделах.

Отклонено сейчас: отдельные section-level функции могут быть тонкими алиасами или presets над общей оберткой приложения, но не самостоятельными pipeline.

### Запретить все прямые импорты из `@shelkovo/markdown` в app

Плюсы:

- максимально простая граница;
- невозможно случайно вызвать package API не тем способом.

Минусы:

- придется проксировать helpers, которые не рендерят body markdown;
- обертка станет свалкой несвязанного API;
- конфиги вроде Astro markdown setup хуже читаются.

Отклонено: риск связан именно с `render` body markdown, а не с низкоуровневыми helpers.

## Последствия

- При добавлении нового Markdown body в `apps/www` разработчик должен импортировать `renderMarkdown` из `@/lib/markdown/render`.
- При добавлении нового доменного preprocessor нужно расширить `RenderSiteMarkdownOptions` и обертку приложения.
- При изменении типографических правил нужно менять `@shelkovo/markdown`, чтобы Markdown-рендер и helpers оставались согласованными.
- `@shelkovo/markdown` должен оставаться свободным от импортов данных и route-утилит сайта.
- Если правило начнут обходить, можно добавить import-boundary test или lint-правило, которое запрещает пакетный `render` вне разрешенной обертки.
- Документация для агентов и README пакета должны явно объяснять разницу между pipeline рендера body markdown и низкоуровневыми helpers.
