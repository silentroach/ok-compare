# ADR-002: Клиентские переходы, предзагрузка по наведению и кеширование HTML

## Статус

Принят

## Дата

2026-05-14

## Контекст

Корневой сайт `kpshelkovo.online` собран как статическое Astro-приложение. В общем макете подключен `<ClientRouter />`, поэтому обычные внутренние HTML-переходы могут выполняться без полной перезагрузки документа. Astro при этом загружает следующий HTML, меняет DOM, запускает события жизненного цикла и завершает навигацию событием `astro:page-load`.

Документация Astro фиксирует несколько важных свойств этой модели:

- `<ClientRouter />` переводит MPA-навигацию в клиентскую навигацию с View Transitions и резервным поведением для браузеров без встроенного View Transition API;
- при использовании `<ClientRouter />` предзагрузка включается по умолчанию с `{ prefetchAll: true }`;
- стратегия `hover` предварительно загружает внутреннюю ссылку при наведении или фокусе, но Astro учитывает режим экономии трафика, медленное соединение и короткие случайные наведения;
- `data-astro-prefetch="false"` является штатным способом отключения предзагрузки для отдельных ссылок;
- `data-astro-reload` является штатным способом отключения клиентской навигации для отдельных ссылок и форм;
- Safari и Firefox чувствительны к заголовкам кеширования для предзагрузки: без `Cache-Control`, `Expires` или `ETag` результат может не переиспользоваться или завершаться ошибкой.

На стороне nginx сайт отдается как набор статических файлов. Значит, политика кеширования nginx является частью пользовательской навигации: если HTML не имеет явных заголовков кеширования, предзагрузка по наведению хуже работает в части браузеров и может давать лишние запросы при последующем клике.

## Решение

Использовать Astro `<ClientRouter />` для всех публичных HTML-страниц сайта.

Явно держать в `astro.config.ts` конфигурацию предзагрузки:

```ts
prefetch: {
  prefetchAll: true,
  defaultStrategy: 'hover',
}
```

Смысл этой конфигурации:

- все обычные внутренние HTML-ссылки являются кандидатами на предзагрузку без ручной разметки каждой ссылки;
- базовая стратегия остается `hover`, потому что она прогревает переход по намерению пользователя, но не делает массовую предзагрузку всех ссылок при загрузке страницы;
- отдельные ссылки могут менять стратегию через `data-astro-prefetch` или отключать предзагрузку через `data-astro-prefetch="false"`.

HTML-страницы должны получать короткий публичный кеш:

```nginx
Cache-Control: public,max-age=60,stale-while-revalidate=300
```

Для HTML также должен оставаться включенным валидатор ответа, например `ETag`, чтобы браузер мог переиспользовать или быстро ревалидировать результат предзагрузки.

Ресурсы сборки под `/static/` считаются отдельным классом. Они получают более длинный кеш и могут использовать `immutable`, потому что имена ресурсов Astro содержат хеш содержимого. Конкретный срок жизни кеша для `/static/` остается операционной настройкой nginx и не фиксируется этим ADR как годовой контракт.

Машиночитаемые адреса, Markdown, ленты, медиа, PDF и другие не-HTML-ресурсы не входят в общий контракт предзагрузки HTML. Для них нужны отдельные политики кеширования по свежести данных и размеру ресурса.

Ссылки того же источника на не-HTML-ресурсы не должны предварительно загружаться как обычные HTML-переходы. Для таких ссылок нужно использовать общий примитив ссылок или явные атрибуты:

- `data-astro-prefetch="false"`, если ресурс не должен прогреваться по наведению;
- `data-astro-reload`, если переход не должен проходить через `<ClientRouter />`;
- `download`, если ресурс является скачиванием;
- `target="_blank"` только когда пользовательский сценарий действительно требует новую вкладку.

Код, который инициализирует DOM-поведение после навигации, должен быть устойчив к клиентской навигации Astro:

- использовать `astro:page-load`, когда нужно запускаться после каждого перехода;
- использовать `astro:after-swap`, когда действие должно произойти сразу после замены DOM;
- быть идемпотентным, потому что встроенные скрипты могут выполняться повторно, а глобальное состояние `window` сохраняется между переходами.

## Область действия

Решение относится к публичным HTML-страницам `apps/www`, которые рендерятся через общий `BaseLayout.astro` и обслуживаются nginx-конфигом `ops/nginx/kpshelkovo-online.conf`.

Решение не фиксирует устаревшие пути ресурсов, временные перенаправления и конкретные сроки жизни кеша для всех типов ресурсов. Они должны развиваться отдельными задачами.

## Рассмотренные альтернативы

### Оставить полную MPA-навигацию

Плюсы:

- меньше клиентского JavaScript;
- стандартное поведение браузера без необходимости думать о событиях жизненного цикла Astro;
- проще для не-HTML ссылок и скачиваний.

Минусы:

- переходы между страницами ощущаются медленнее;
- нельзя использовать View Transitions и общий навигационный индикатор прогресса без полной перезагрузки;
- предзагрузка по наведению перестает быть частью единой навигационной модели.

Отклонено: для текущего сайта важнее быстрые переходы между статическими HTML-страницами и плавная пользовательская навигация.

### Включить предзагрузку только на явно помеченных ссылках

Плюсы:

- меньше риск случайно предварительно загружать не-HTML-ресурсы;
- точнее контроль расхода трафика.

Минусы:

- нужно вручную размечать каждую навигационную ссылку;
- новые разделы легко забывают получить предзагрузку;
- `<ClientRouter />` уже задает поведение по умолчанию вокруг `prefetchAll: true`.

Отклонено сейчас: глобальная предзагрузка по наведению лучше соответствует сайту с большим количеством статических HTML-переходов. Риск не-HTML-ссылок закрываем отдельным контрактом примитива ссылок на ресурсы.

### Использовать `viewport` или `load` как стратегию по умолчанию

Плюсы:

- переходы могут быть еще быстрее, потому что страницы прогреваются раньше клика;
- `viewport` может хорошо работать для коротких меню и карточных списков.

Минусы:

- больше фоновых запросов на страницах со многими ссылками;
- выше риск предзагрузки нецелевых страниц;
- `load` особенно агрессивен и плохо соответствует контентным страницам с большим числом ссылок.

Отклонено: `hover` лучше балансирует скорость и стоимость сети.

### Использовать только `tap`

Плюсы:

- минимальный лишний трафик;
- меньше риска предварительно загружать большие ресурсы.

Минусы:

- прогрев начинается слишком поздно и почти не улучшает навигацию по наведению на компьютере;
- теряется основная польза предзагрузки для страниц, на которые пользователь явно навел курсор.

Отклонено: `tap` остается запасным вариантом для медленного соединения и режима экономии трафика, но не основной стратегией.

### Включить экспериментальный клиентский предрендеринг

Плюсы:

- потенциально еще быстрее в Chromium через Speculation Rules;
- можно управлять жадностью предрендеринга для отдельных сценариев.

Минусы:

- экспериментальная функция Astro;
- выше стоимость CPU, памяти и сети;
- текущая задача решается обычной предзагрузкой без предрендеринга.

Отклонено до отдельной необходимости и измерений.

### Держать HTML без публичного кеша

Плюсы:

- максимальная свежесть после деплоя;
- меньше риска показать старую страницу.

Минусы:

- предзагрузка хуже переиспользуется в Safari/Firefox;
- больше запросов к nginx;
- статический сайт не использует преимущество дешевой краткой ревалидации.

Отклонено: короткий публичный кеш с `stale-while-revalidate` лучше для статических HTML-страниц и предзагрузки.

## Последствия

- Новые семейства HTML-маршрутов должны получать явную политику кеширования HTML в nginx или другой обслуживающей инфраструктуре.
- Новые ссылки на не-HTML-ресурсы того же источника должны идти через общий примитив ссылок на ресурсы или явно отключать предзагрузку.
- Скрипты, которые зависят от DOM, должны учитывать жизненный цикл Astro ClientRouter.
- Изменения поведения деплоя, `base` и корня для `apps/www` должны синхронно проверять nginx-конфиг.
- Встроенные в браузер междокументные View Transitions стоит пересмотреть позже, когда они станут достаточной заменой `<ClientRouter />` для наших сценариев.
